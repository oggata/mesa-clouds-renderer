# 第14章　Isaac SimによるReal-to-Simシミュレーション
---

第2部までで、実機のSO-101で模倣学習を1周できるようになりました。第3部では、**シミュレーション**を使ってデータを補い、実機の負担を減らしながら性能とロバスト性を高める世界に入ります。その中核が、NVIDIAの物理シミュレーター **Isaac Sim** です。

## 14.1 Isaac Sim / Isaac Labとは

**Isaac Sim** は、物理ベースの仮想環境でロボティクスを設計・シミュレート・テスト・トレーニングするためのフレームワークで、NVIDIA Omniverse上に構築され、OpenUSD形式を採用しています。NVIDIA PhysXによる高精度な物理シミュレーション、フォトリアルなレイトレーシング、複数環境の並列シミュレーション、強化学習・模倣学習との統合が特徴です。最小要件はWindows 11・RTX 3060・VRAM 6GBほどで、[公式サイト](https://docs.isaacsim.omniverse.nvidia.com/latest/installation/quick-install.html)からインストールできます。

![画像](https://static.zenn.studio/user-upload/fe16b91df571-20260129.gif)

Isaac Simの主要ワークフローは3つあります。①**合成データ生成**（実世界での収集が難しいデータをシミュレーションで生成）、②**ロボットスタックの検証**（SIL/HILテスト）、③**ロボット学習**（Isaac Labを通じた強化学習・模倣学習）です。SO-101のようなマニピュレーターに加え、ヒューマノイド・AMR・ドローンなど多様なロボットに対応します。

**Isaac Lab** は、Isaac Sim上に構築されたロボット学習のための統一フレームワークで、環境管理・マルチ環境並列化・センサー抽象化・タスク定義といった高レベルAPIを提供します。両者の関係は次の階層になります。

```
ロボット学習アプリケーション（強化学習／模倣学習／VLA）
        ↓
Isaac Lab（環境管理・並列化・センサー抽象化・タスク定義）
        ↓
Isaac Sim（物理シミュレーション PhysX・レンダリング・アセット管理）
```

:::message
📷 **画像プレースホルダー**：Isaac Sim / Isaac Lab / 学習アプリケーションの3層アーキテクチャ図。各層の役割（物理・レンダリング／環境管理・並列化／RL・IL・VLA）を明記し、SO-101がどこに位置づくかを示す。
:::

Isaac LabでSO-101を扱う際は、ロボット本体（`ArticulationCfg` でUSDアセットとアクチュエータを定義）、対象物体、カメラ・力覚などのセンサー、そして観測・報酬・終了条件を組み合わせて環境を構成し、1,000環境規模での並列シミュレーションが可能です。

## 14.2 SO-101とIsaac Simの連携プロジェクト

SO-101とIsaac Simの連携には、すでに実用的なプロジェクトがいくつも存在します。代表的な3つを押さえておきましょう。

| プロジェクト | 開発 | 主な機能 | 用途 |
|---|---|---|---|
| **isaac_so_arm101** | MuammerBay | Isaac Lab上でのReach/PickAndPlaceタスク実装、PPO強化学習 | シミュレーション内での強化学習 |
| **LeIsaac** | Seeed Studio | 実機リーダーでシミュを操作しデータ収集、LeRobot連携 | Sim内でのテレオペ・データ収集 |
| **Real-to-Sim Twin** | Vipin | ROS2で実機とIsaac Simをリアルタイム同期 | デジタルツイン・双方向同期 |

**isaac_so_arm101** は、`SO-ARM101-Reach-Play-v0` や `SO-ARM101-PickAndPlace-Play-v0` といったタスク環境を提供し、ゼロアクション／ランダムアクションでの動作テストから、PPO（強化学習ライブラリRSL-RL）による1,024環境並列トレーニングまでを行えます。

![画像](https://static.zenn.studio/user-upload/c6ee2985890f-20260121.png)

**LeIsaac** は、実機のリーダーアームでIsaac Lab内のフォロワーを操作してデータを収集し、それをLeRobotフォーマット（第10章で見た `observation.state`／`observation.images`／`action` 等）に変換します。実機だけでは集めにくい多様な初期条件のデータを、シミュレーションで安全・大量に生成できるのが強みです。

![画像](https://static.zenn.studio/user-upload/bee71c7e8f97-20260121.png)

**Real-to-Sim Twin** は、ROS2（Humble）とCycloneDDSを使い、実機の関節状態をトピックとしてパブリッシュしてIsaac Sim内のロボットに反映し、逆にシミュレーションで生成したコマンドを実機に送る双方向のデジタルツインです。実機とシミュの関節名の不一致や、ネットワーク上の他ロボットのトピック混入（`ROS_DOMAIN_ID` で分離）といった実務的なハマりどころも、このプロジェクトから学べます。

![画像](https://static.zenn.studio/user-upload/a2dd117f9dfd-20260121.png)

## 14.3 GR00T N1.5との統合（概要）

**NVIDIA Isaac GR00T N1.5** は、汎用ロボットの推論とスキル習得のためのオープンな**VLA（Vision-Language-Action）基盤モデル**で、言語と画像のマルチモーダル入力を処理し、SO-101との統合が公式にサポートされています。Isaac Lab（LeIsaac）で集めたデータをLeRobotフォーマットに変換し、GR00T N1.5を自分のタスクでファインチューニングし、推論サーバー（GPU上でモデル実行）と制御クライアント（実機通信）を分離した構成でデプロイする、という流れです。GR00Tは自然言語指示（「オレンジを掴んで皿に置いて」など）で制御でき、VLA基盤モデルの詳細は**第18章**で扱います。

## 14.4 Real-to-Sim-to-Realの実践ワークフロー

実機とシミュレーションを組み合わせた典型的なパイプラインは、次の4ステップです。

1. **実機でデモ収集**：LeRobotで50エピソード程度の実機デモを収集（第8章）
2. **シミュで拡張**：実機データをIsaac Labにインポートし、多様な初期条件・外乱でシミュレーションデータを大量生成（LeIsaac）
3. **統合して学習**：実機データとシミュレーションデータを統合し、ACTやGR00Tで学習
4. **Sim評価とReal評価の反復**：シミュレーションで大量評価（安全・高速）し、実機で最終確認して、両者の差（Sim-to-Realギャップ）を測りながら改善

## 14.5 Sim-to-Realギャップを埋める

「シミュレーションでは成功するのに実機では失敗する」——これがSim-to-Realギャップです。埋めるための主な手立ては3つです。

**物理パラメータの調整** — シミュレーションのアクチュエータの剛性（stiffness）・減衰（damping）・摩擦（friction）を、実機の挙動に合わせます。

**Domain Randomization** — 関節摩擦・質量・照明などをランダムに変動させて学習することで、特定のシミュ設定に過学習せず、実機の「見たことのない条件」にも対応できる汎用性を獲得します（第1部・第12章でも触れた考え方です）。

**センサーノイズの追加** — シミュレーションのカメラにガウシアンノイズを加えるなど、理想的すぎるセンサー出力を現実に近づけます。

これらを組み合わせることで、シミュレーションで学んだポリシーが実機でも通用しやすくなります。

## 14.6 トラブルシューティング

| 症状 | 対処 |
|---|---|
| Isaac SimのFPSが低い | 不要な拡張を無効化、反射・透過などレンダリング設定を軽量化 |
| Simで成功・実機で失敗 | 物理パラメータを実機に合わせる、Domain Randomizationを導入、センサーノイズを追加 |
| テレオペの遅延が大きい | CPU親和性・リアルタイム優先度の設定、ROS2のQoS（CycloneDDS）を最適化 |
| ROS2で余計なトピックが混入 | `ROS_DOMAIN_ID` で独自ドメインに分離 |

## 14.7 まとめと学習パス

本章では、Isaac Sim / Isaac Labの基礎、SO-101との連携プロジェクト3種（isaac_so_arm101・LeIsaac・Real-to-Sim Twin）、GR00T N1.5統合の概要、そしてReal-to-Sim-to-RealのワークフローとSim-to-Realギャップの埋め方を見ました。

学習パスとしては、まず**初級**でIsaac Simの基本操作とisaac_so_arm101のダミー/ランダムエージェント実行に慣れ、**中級**でLeIsaacによるテレオペ・データ収集とカスタムタスク作成を体験し、**上級**で強化学習ポリシーの訓練・GR00Tのファインチューニング・実機デプロイとSim-to-Realギャップの最小化に取り組む、という順序がおすすめです。

次章では、Isaac Simと連携する生成AIワールドモデル **NVIDIA Cosmos** を取り上げ、シミュレーションデータをさらに豊かにする方法を見ていきます。

# 第15章　Isaac SimとNVIDIA Cosmosの統合関係
---

前章のIsaac Simは「物理的に正確なシミュレーション」を担いますが、視覚的な多様性やリアリズムには限界があります。それを補うのが、生成AIによるワールドモデル **NVIDIA Cosmos** です。本章では、両者の役割分担と統合ワークフローを見ていきます。

## 15.1 Isaac SimとCosmosの役割分担

NVIDIAのPhysical AI開発は、Isaac SimとCosmosを両輪とするエコシステムで構成されます。Isaac Simがロボット制御コマンドから物理的に正確なセンサーデータを生成し、Cosmosがそのデータを視覚的にリアルな多様なデータへと拡張し、両者の成果をIsaac Lab / GR00Tでのロボット学習に流し込む、という流れです。

| 観点 | Isaac Sim | NVIDIA Cosmos |
|---|---|---|
| 主な目的 | ロボットシミュレーション | 物理世界のAIモデル（World Model） |
| 入力 | ロボット制御コマンド | 画像・動画・言語 |
| 出力 | センサーデータ・物理状態 | 予測動画・拡張データ |
| 強み | 物理的正確性 | 視覚的リアリズム |
| 使用場面 | 学習・テスト・検証 | データ拡張・予測 |

Isaac Simは物理的に正確ですが、視覚的多様性が不足し、Domain Randomizationにも手間がかかります。この弱点をCosmosが補完します。

:::message
📷 **画像プレースホルダー**：Isaac SimとCosmosの役割分担図。「Isaac Sim（物理的正確性）」と「Cosmos（視覚的リアリズム）」が双方向にデータをやり取りし、Isaac Lab/GR00Tの学習に統合される様子を示す。両者が補完関係にあることが伝わる図。
:::

## 15.2 Cosmosの3つのモデル

**Cosmos** は、Physical AI開発のためのWorld Foundation Modelプラットフォームで、3つのモデルからなります。

![画像](https://static.zenn.studio/user-upload/723d57db8ad8-20260120.gif)

- **Cosmos-Predict（予測）**：現在の観測・ロボット状態・提案アクションから、未来のフレーム（例：1秒先の30フレーム）を予測生成します。行動を実行する前に「この行動を取ったらどうなるか」を予測し、衝突チェックなどに使えます。
- **Cosmos-Transfer（転移）**：シミュレーション画像を実世界風のリアルな画像に変換します。幾何形状を保ったまま、照明や天候、質感を変えたバリエーションを生成でき、データ拡張の要になります。
- **Cosmos-Reason（推論）**：言語指示（「オレンジのカップを掴んで棚に置いて」）と視覚観測から、段階的なアクションプラン（「カップを探す→近づく→掴む→棚へ移動→置く」）を生成します。シーン内の物体の3D位置やバウンディングボックスを構造化して出力することもできます。

![画像](https://static.zenn.studio/user-upload/628d95cad77a-20260121.png)

## 15.3 統合ワークフロー：Cosmosによるデータ拡張

Isaac SimとCosmosを組み合わせる典型的なパターンは、**「少量のベースデータをCosmosで大量に拡張する」**ことです。Isaac Simで50エピソードのベースデータを収集し、Cosmos-Transferで各エピソードの画像を照明（朝／昼／夕）や天候（晴／曇／雨）を変えて変換すれば、10倍（500エピソード）のデータセットになります。さらにCosmos-Predictで「予測した次フレームと実際のフレームの差」を測れば、物理的に妥当でない拡張データをフィルタリングして品質を保てます。

:::message
📷 **画像プレースホルダー**：Cosmosによるデータ拡張パイプライン図。「Isaac Simで50エピソード収集 → Cosmos-Transferで照明・天候を変えて10倍に拡張 → Cosmos-Predictで物理的妥当性を検証 → 学習」という流れを、拡張前後のサンプル画像とともに示す。
:::

## 15.4 GR00T-MimicとGR00T-Dreams

Cosmos関連の技術として、データ生成に特化した2つのモデルがあります。**GR00T-Mimic** は、少数の実演データ（例：10個）から多数の合成バリエーション（例：200個）を生成し、実機デモの水増しに使えます。**GR00T-Dreams** は、テキスト記述（「カップを掴んで棚に置く」）とエンボディメント（so101）から動画を生成し、その動画からアクションを抽出します。実機もシミュレーションも使わず、「想像」からデータを作り出すアプローチです。

## 15.5 統合のメリットとSim-to-Real

Isaac SimとCosmosの統合により、**データ効率**（10エピソードの実機データから1,000エピソード規模のデータセットを生成）、**安全性**（シミュレーションで事前検証）、**コスト削減**（実機実験の最小化）が得られます。

Sim-to-Realギャップを埋めるには、前章と同様、Domain Randomization（関節摩擦・質量・物体特性をランダム化）と、実機データの混合（例：シミュレーション70%＋実機30%）が有効です。GPUメモリが不足する場合は、CosmosのバッチサイズやIsaac Simの並列環境数を減らし、Mixed Precisionを使うと軽減できます。

## 15.6 まとめ

本章では、Isaac SimとNVIDIA Cosmosの統合を見ました。要点は、**Isaac Simが物理的正確性を、Cosmosが視覚的リアリズムを担い、互いを補完する**こと、**Cosmosの3モデル（Predict/Transfer/Reason）で予測・データ拡張・言語推論ができる**こと、そして**少量の実機データをCosmosで大量に拡張することでデータ効率が飛躍的に高まる**ことです。物理的に正確なシミュレーションとAI駆動のデータ生成を組み合わせることで、効率的なPhysical AI開発が可能になります。

次章では、より軽量で個人でも扱いやすいシミュレーター **Genesis** を使い、模倣学習から並列強化学習、Sim-to-Realまでを一気通貫で体験します。

# 第16章　GenesisによるSim-to-Real — 模倣学習から並列強化学習まで
---

前章までのIsaac Sim / CosmosはNVIDIA GPUを前提とする本格的な環境でした。本章では、より軽量で**Macでも動く**シミュレーター **Genesis** を使い、「コントローラー操作 → 模倣データ収集 → 並列強化学習（PPO）→ 実機転移（Sim-to-Real）」という一連のパイプラインを、筆者公開の [genesis-so101-arm-demo](https://github.com/oggata/genesis-so101-arm-demo) で体験します。

## 16.1 Genesisとは

**Genesis**（genesis-world）は、2024年末にリリースされたPython製の物理シミュレーターです。PyTorchバックエンドでGPU並列化に対応し、数千の環境を同時実行できます。従来のロボット強化学習はMuJoCoやIsaac Gymが主流でしたが、Genesisはセットアップが圧倒的に軽量で、**Macのローカル環境でも動作する**のが大きな特徴です。1行で数百〜数千の環境を並列実行でき、`pip install genesis-world` で導入でき、URDFの既存ロボットモデルをそのまま使え、OpenCV/OpenGLでリアルタイムに可視化できます。

![画像](https://static.zenn.studio/user-upload/cf7da9171669-20260423.gif)

## 16.2 デモの全体構成

デモ（`genesis-so101-arm-demo`）は、メインスクリプト `so101_mac_viewer.py`、Gymnasium準拠のSO-101環境クラス `so101.py` などからなり、3つの動作モードを持ちます。

| モード | 起動コマンド | 説明 |
|---|---|---|
| 手動操作 | `python so101_mac_viewer.py` | キーボードでジョイントを操作 |
| 学習 | `python so101_mac_viewer.py --mode train` | PPOで強化学習を実行 |
| デモ | `python so101_mac_viewer.py --mode demo` | 学習済みモデルを動かす |

## 16.3 ステップ1：キーボード操作でデモ収集

引数なしで起動するとキーボード操作モードになります。

```bash
pip install genesis-world stable-baselines3 gymnasium opencv-python
python so101_mac_viewer.py
```

F1〜F10でベース回転・ショルダー・エルボー・リストの各ジョイントを、`-`/`=` でグリッパーを操作します。画面にはオーバーヘッドカメラとリストカメラの2ビューが表示され、キューブを掴む動作を手動で行って軌跡データを蓄積します。これが後の模倣学習の初期ポリシーになります。

## 16.4 ステップ2：Genesisによる並列強化学習

`so101.py` にはGymnasium準拠の環境クラスが実装されています。Genesisでは `num_envs` を増やすだけで並列化が有効になります（デフォルトは4並列、GPUがあれば数十〜数百に拡張可能）。

報酬設計はシンプルで、エンドエフェクタとキューブの距離が縮まるほど高報酬とし、キューブを持ち上げたらボーナスを加える、という形です。

```python
def compute_reward(self):
    ee_pos = self.robot.get_link_pos('gripper_link')
    cube_pos = self.cube.get_pos()
    dist = torch.norm(ee_pos - cube_pos, dim=-1)
    reward = -dist                    # 距離が縮まるほど高報酬
    if cube_pos[2] > 0.05:            # 持ち上げたらボーナス
        reward += 1.0
    return reward
```

PPO学習は次のコマンドで実行し、完了すると `best_model.zip` などが保存されます。

```bash
python so101_mac_viewer.py --mode train --num-envs 4 --total-timesteps 60000
```

📷 **画像プレースホルダー**：Genesisの並列環境で複数のSO-101が同時に強化学習している様子のスクリーンショット。「1つのGPU/CPUで多数の環境が並列に試行している」という並列強化学習の利点が伝わる図。
:::

## 16.5 ステップ3：デモモードで動作確認

```bash
python so101_mac_viewer.py --mode demo
```

学習済みモデルがロードされ、シミュレーター内でキューブ把持動作が自律実行されます。動作が安定していれば、実機への転移（Sim-to-Real）に進む準備が整っています。

## 16.6 Sim-to-Realの課題とアプローチ

シミュレーターで学習したポリシーをそのまま実機に移してもうまくいかないことが多く、この「シミュレーションと現実のギャップ」を **Reality Gap** と呼びます。主な原因は、物理パラメータのズレ（摩擦・質量・慣性）、センサーノイズ（実機カメラの歪み）、アクチュエーター遅延（モーター応答速度の違い）です。

対策として、このデモでは物体の物理パラメータ（摩擦・弾性・密度）をコードから実機に合わせて調整できます。加えて、学習時に摩擦や質量をランダムに変える **Domain Randomization** により、実機のばらつきに頑健なポリシーが得られます（第14章・第15章と共通の考え方です）。

|  | Genesis | Isaac Gym | MuJoCo |
|---|---|---|---|
| セットアップ | pip一発 | NVIDIA GPU必須 | ライセンス必要 |
| Mac対応 | ○ | ✗ | ○ |
| 並列化 | ○（GPU/CPU） | ○（GPU必須） | △ |
| URDF対応 | ○ | ○ | ○ |

GenesisがMacでも動く点は、個人開発者にとって大きなメリットです。高価なGPUサーバーがなくても、ローカルでロボット強化学習の開発サイクルを回せます。

## 16.8 まとめ

本章では、genesis-so101-arm-demoによるSim-to-Realパイプラインを体験しました。キーボード操作でシミュレーター内のSO-101を動かして感覚を掴み、Genesisの並列環境でPPO強化学習を高速に回し、デモモードで動作確認して実機転移に備える——という流れです。Genesisの軽量さとMac対応は、**実機を持っていない段階でも学習パイプラインを試せる**という点で価値があります。実機を持っている方は、学習したポリシーをLeRobotのインターフェース経由でSO-101に転送すれば、完全なSim-to-Realループを構築できます。

# 第17章　シミュレーターと物理環境の接続 — World Modelsから実機制御へ
---

前章までで、Isaac Sim・Cosmos・Genesisという3つのシミュレーション環境を見てきました。本章では、それらの「World Model（世界モデル）」と物理ロボットSO-101をどう接続するか——すなわち **Sim-to-Realの橋渡し** を、3つのアプローチとして整理します。

## 17.1 「美しいビデオ」と「動くロボット」のギャップ

Cosmosは驚くほどリアルな動作予測ビデオを生成できます。しかし「クッキーを掴んで」という予測ビデオが得られても、それだけではSO-101は動きません。**ロボットを動かすには、関節角度を出力する「ポリシー」が必要**であり、リアルな動画とポリシーの間には大きなギャップがあります。

![画像](https://static.zenn.studio/user-upload/723d57db8ad8-20260120.gif)

このギャップの正体は、出力形式の違いにあります。

| 項目 | World Model（Cosmos等） | SO-101制御に必要なもの |
|---|---|---|
| 出力 | RGB画像（H×W×3） | 関節角度（7自由度） |
| 座標系 | ピクセル空間 | 物理座標（ラジアン/メートル） |
| 物理量 | 推定・学習ベース | センサーからの実測値 |
| 時間精度 | フレームレート依存 | リアルタイム制御（〜100Hz） |

このギャップを埋める方法を、3つのアプローチとして見ていきます。

:::message
📷 **画像プレースホルダー**：World Modelの出力（RGB動画）と実機制御に必要なもの（関節角度）のギャップを示す図。「ピクセル空間 → 物理座標への変換が必要」という橋渡しの必要性を、Cosmosの出力とSO-101の関節を対比して示す。
:::

## 17.2 アプローチ1：物理シミュレーターとのハイブリッド

最も確実なのは、**Cosmosで戦略を評価し、Isaac Labで正確な物理量を得て学習する**ハイブリッドです。World Modelは物理量を推定するだけですが、Isaac Labのような物理シミュレーターは関節角度・速度・力を正確に取得できます。

具体的には、①Cosmosで複数の戦略（「上から接近」「横から接近」「回転しながら接近」）を予測して成功しそうなものを選び、②その戦略をIsaac Labで実行して正確な関節角度・速度を含むデータを収集し、③そのデータでLeRobotのACTポリシーを学習し、④実機に転送する、という流れです。Cosmosが「計画（どの戦略が良いか）」を、Isaac Labが「正確な物理量」を担う分業になります。

## 17.3 アプローチ2：ビジョンベースの状態推定

World Modelの出力ビデオから、**画像処理で物理量を推定する**アプローチです。SAM（物体セグメンテーション）で対象物のマスクを取り、FoundationPoseで6自由度の姿勢を推定し、MediaPipe等でアームのキーポイントを検出して逆運動学で関節角度を逆算します。これらをLeRobotデータセット形式に変換すれば、World Modelの予測だけから学習データを作れます。

ただし、この推定には無視できない誤差が伴います。

| 推定項目 | 典型的な誤差 | 影響 |
|---|---|---|
| 関節角度 | ±5〜10度 | グリッパー位置が最大2cmずれる |
| 物体姿勢 | ±2cm・±15度 | 把持失敗の可能性 |
| 深度情報 | ±3〜5cm | 衝突リスク |

対策として、実機での微調整（Residual Learning）や、安全マージンを持たせたアクション生成、さらにArUcoマーカーをアームに貼って推定精度を上げる、といった方法があります。既存の動画データを活用したい場合に有効ですが、精度面では次のアプローチ1に劣ります。

## 17.4 アプローチ3：強化学習による自動適応

シミュレーションで学習したポリシーを、**実機で強化学習によって微調整する**アプローチです。Isaac Labなどで学習したベースポリシーはそのまま使い、その出力に対する小さな補正を実機環境での強化学習（Residual RL）で学ばせます。「ベースの行動 ＋ 少量の補正」という形にすることで、実機のばらつきに合わせた適応を、ゼロから学習し直すより遥かに少ないエピソードで達成できます。

このとき、学習側でIsaac LabにCosmosのような視覚的多様性を持たせる **Domain Randomization**（照明の強さ・色温度、カメラ位置のノイズ、テクスチャ、質量・摩擦のスケールをランダム化）を併用すると、実機の見た目や物理のばらつきに頑健なポリシーが得られます。

## 17.5 3アプローチの使い分け

推奨される全体像は、**「Cosmosで計画 → Isaac Labで訓練 → 実機で適応・実行」** の流れです。各アプローチはシナリオによって使い分けます。

| シナリオ | 推奨手法 | 理由 |
|---|---|---|
| 新タスクの開発 | Cosmos + Isaac Lab（アプローチ1） | 安全に試行錯誤できる |
| 高精度が必要 | Isaac Lab単体 | 物理量が正確 |
| データが少ない | Cosmosのファインチューニング | 少ないデモから学習 |
| 既存の動画データのみ | ビジョンベース推定（アプローチ2） | 既存データを活用できる |
| 実機との差を詰めたい | 強化学習による適応（アプローチ3） | Reality Gapを実機で吸収 |

:::message
📷 **画像プレースホルダー**：3アプローチの比較図。「①物理シミュレーターとのハイブリッド」「②ビジョンベース状態推定」「③強化学習による自動適応」を、精度・データ要件・実装難易度の観点で並べ、使い分けの指針を示す。
:::

## 17.6 トラブルシューティング

**Isaac Labの出力が実機と大きく異なる**場合は、シミュと実機の軌跡の差（位置誤差・タイミング誤差・力の差）を分析し、位置誤差が5cm以上ならURDFの質量・慣性を、力の差が大きければ摩擦係数を調整します。**Cosmosの予測が不正確**な場合は、SO-101の実データでCosmosをファインチューニングします。**ビジョンベース推定の精度が足りない**場合は、ArUcoマーカーによる補正を追加すると逆運動学の精度が上がります。

## 17.7 まとめ

本章では、World Modelと実機SO-101を接続する3つのアプローチ——物理シミュレーターとのハイブリッド、ビジョンベースの状態推定、強化学習による自動適応——を整理しました。要点は、**World Modelの美しい予測をそのまま実機に使うことはできず、関節角度という物理量への橋渡しが必要**なこと、そして**「Cosmosで計画 → Isaac Labで訓練 → 実機で適応」という分業が最も確実**なことです。精度・データ要件・手間のトレードオフを見極めて、シナリオに応じたアプローチを選びましょう。

次章では、これらのシミュレーション技術とも結びつく最先端の潮流——π0やGR00Tといった **VLA（Vision-Language-Action）基盤モデル** を取り上げ、本書の締めくくりとします。

# 第18章　VLA基盤モデルの活用 — π0 / GR00T のfine-tune
---

本書の締めくくりとして、2024〜2025年のロボット学習で最も注目されている潮流——**VLA（Vision-Language-Action）基盤モデル**——を取り上げます。ここまで学んできたACT（第11章）が「1つのタスクを1から学ぶ」手法だったのに対し、VLAは「あらゆるタスクの土台となる大規模な事前学習済みモデルを、自分のタスクに少しだけ適応させる」という、まったく異なるアプローチです。

## 18.1 VLA基盤モデルとは

**VLA（Vision-Language-Action）モデル**は、視覚（Vision）・言語（Language）・行動（Action）を統合的に扱うモデルです。カメラ画像と自然言語の指示を入力として、ロボットの行動を直接出力します。「オレンジを掴んで皿に置いて」という言葉で指示でき、訓練していないタスクにもある程度対応できる汎用性が最大の特徴です。

LLM（大規模言語モデル）が「インターネット規模のテキストで事前学習し、少数の例でタスクに適応する」ように、VLAは「多様なロボット・多様なタスクの大規模データで事前学習し、少数のデモで自分のロボット・タスクに適応する」——**基盤モデル（Foundation Model）のパラダイムをロボティクスに持ち込んだもの**です。

:::message
📷 **画像プレースホルダー**：ACT（タスクごとに1から学習）とVLA基盤モデル（大規模事前学習＋少数デモでfine-tune）のアプローチの違いを対比する図。VLAが「言語指示＋画像 → 行動」を統合的に扱う様子と、事前学習の規模感を示す。
:::

## 18.2 主要なVLA基盤モデル

**π0（Pi Zero）** は、Physical Intelligence社が開発したVLA基盤モデルです。多様なロボットの大規模データで事前学習されており、フローマッチングという手法で滑らかな行動を生成します。SO-101のような小型アームから両腕ロボットまで、幅広いエンボディメントに対応します。コンパクトな実装の **SmolVLA** はLeRobotに統合されており、より手軽に試せます。

**NVIDIA Isaac GR00T N1.5**（第14章でも触れました）は、汎用ロボット向けのオープンなVLA基盤モデルで、言語と画像のマルチモーダル入力を処理し、クロスエンボディメント（複数の異なるロボットへの転移）に対応します。SO-101との統合が公式にサポートされ、Isaac Lab経由でのデータ収集からファインチューニング、実機デプロイまでの一連のワークフローが整備されています。

**Hi Robot**（第12章）は、階層的なVLAで、高レベルの推論（複雑な指示を原子的コマンドに分解）と低レベルの実行を分離し、「ベジタリアンサンドイッチを作って」のような複雑な指示に対応します。

## 18.3 なぜVLAが重要なのか

ACTのような手法は、タスクごとに数十〜数百のデモを集めて1から学習する必要がありました。VLA基盤モデルは、すでに膨大なデータで「物を掴む」「運ぶ」「置く」といった基本的なスキルや、言語と行動の対応関係を学習済みです。そのため、自分の特定タスクには**少数のデモでファインチューニングするだけ**で高い性能に到達できます。

この違いは、画像認識でImageNetの事前学習モデルをファインチューニングする、あるいは自然言語処理でBERTやGPTをファインチューニングするのと同じ発想です。ロボティクスにも「事前学習された土台の上に、少しの追加学習で自分の課題を解く」時代が来ています。

## 18.4 SO-101でVLAを使う実践フロー

SO-101でVLA基盤モデル（GR00Tを例に）を活用する典型的な流れは次の通りです。

1. **データ収集**：実機テレオペ（第8章）またはIsaac Lab/LeIsaac（第14章）で、自分のタスクのデモを集め、LeRobotフォーマットに変換します。VLAは基盤モデルのおかげで、ACTより少ないデモ数でも成立しやすいのが利点です。
2. **設定ファイルの準備**：SO-101のエンボディメント（行動次元＝6関節＋グリッパー、カメラ構成、各関節の可動範囲）を設定します。
3. **ファインチューニング**：事前学習済みのGR00T N1.5を、自分のデータで追加学習します。GPUが必要なため、クラウドGPUを使うのが現実的です。
4. **評価**：シミュレーション（Isaac Lab）で大量評価し、成功率・エピソード長・衝突率・動作の滑らかさを測ります。
5. **実機デプロイ**：GR00Tは**推論サーバー（GPU上でモデル実行）と制御クライアント（実機通信）を分離**したアーキテクチャを採用しています。推論サーバーをGPUマシンで立て、制御クライアントがSO-101と通信しながら、自然言語指示（「オレンジを掴んで皿に置いて」）で制御します。

```bash
# 推論サーバー（GPUマシン）— 概念的なコマンド
python scripts/inference_service.py --server \
  --model_path ./so101-checkpoints/last \
  --embodiment_tag new_embodiment --port 5555

# 制御クライアント（実機側）
python scripts/deploy_real_robot.py \
  --policy_host localhost --policy_port 5555 \
  --robot_port <SO-101のポート> \
  --policy_language_instruction "Pick up the orange and place it on the plate"
```

:::message
📷 **画像プレースホルダー**：GR00Tのデプロイ構成図。「推論サーバー（GPU上でVLAモデル実行）」と「制御クライアント（SO-101と通信）」がTCP/IPで接続され、自然言語指示で実機が動く様子を示す。推論と制御の分離が伝わる図。
:::

## 18.5 VLAの課題と展望

VLAは強力ですが、課題もあります。**計算コスト**——大規模モデルの推論にはGPUが必要で、リアルタイム制御には推論の高速化（denoising stepsの削減など）が求められます。**データフォーマットの整合**——エンボディメントごとに行動空間やカメラ構成が異なるため、設定を正しく合わせる必要があります。**Sim-to-Realギャップ**——シミュレーションデータで学習した場合、これまでの章と同じくDomain Randomizationや実機データの混合が有効です。

それでも、VLA基盤モデルは「1つのモデルが多様なタスク・多様なロボットをこなす」という汎用ロボットの夢に最も近いアプローチです。SO-101のような手頃なハードウェアで、こうした最先端の基盤モデルをファインチューニングして試せること自体が、LeRobotエコシステムがもたらした大きな変化です。

## 18.6 まとめ

本章では、VLA（Vision-Language-Action）基盤モデルを取り上げました。要点は、**VLAは視覚・言語・行動を統合し、言語指示でロボットを制御できる**こと、**基盤モデルのパラダイム（大規模事前学習＋少数デモでのfine-tune）をロボティクスに持ち込んだ**こと、そして**π0やGR00T N1.5をSO-101で少数デモからファインチューニングできる**ことです。ACTのような1タスク特化の手法と、VLAのような汎用基盤モデルを使い分けることで、SO-101でのロボット学習の可能性は大きく広がります。

次章では、このVLA基盤モデルの最新版である **π0.5** を取り上げ、コミュニティが公開している実際のファインチューン成果物を使って、SO-101を動かす具体的な手順を見ていきます。

# 第19章　π0.5でSO-101を動かす — コミュニティのファインチューン成果物を使う
---

前章では、VLA基盤モデルの概念と、π0・GR00Tのファインチューンを俯瞰しました。本章では、その最新版である **π0.5** を実際にSO-101で動かす方法を、**すでにコミュニティが公開しているファインチューン成果物**を手がかりに具体的に見ていきます。

## 19.1 π0.5とは

**π0.5** は、Physical Intelligence社が開発したVLA基盤モデル **π0** の後継にあたるモデルです。前章で見た通り、VLAは視覚・言語・行動を統合し、大規模データで事前学習された「土台」を少数のデモで自分のタスクに適応させる——という基盤モデルのパラダイムに立っています。π0.5はこの系譜の中で、より広いタスク・環境への汎化を狙ったバージョンとして位置づけられます。

LeRobotのエコシステムでは、Physical Intelligenceの公式ベースモデル **`lerobot/pi05_base`** が公開されています。ただし重要なのは、**この公式ベースモデルはSO-101専用ではなく汎用のベースモデル**だという点です（LIBERO/DROID向けの派生も公式に提供されています）。つまり、`pi05_base` をそのまま読み込んでもSO-101が思い通りに動くわけではなく、**SO-101のデータでファインチューンされたポリシー**が別途必要になります。

:::message
📷 **画像プレースホルダー**：π0.5の位置づけを示す図。「公式の汎用ベースモデル `lerobot/pi05_base`」→「SO-101データでファインチューン」→「SO-101で動くポリシー（コミュニティ公開）」という派生の流れを、GR00T/π0など他の基盤モデルと並べて示す。
:::

## 19.2 「π0.5のSO-101ポリシー」はコミュニティの成果物

ここが本章のポイントです。**「π0.5でSO-101を動かすポリシー」は、Physical Intelligenceの公式リリースではなく、コミュニティのユーザーがファインチューンして公開した成果物**という位置づけです。すでにHugging Face上に、SO-101向けにπ0.5をファインチューンして公開している実例があります。

**`felixmayor/pi05_so101_orange_cube`** は、π0.5をSO-101アーム向けにファインチューンしたポリシーです。オレンジのキューブを掴んで置く（pick & place）タスクを、**154エピソード / 68.5kフレーム**のデータで、5000ステップ学習し、最終的な loss は 0.0315 という結果が公開されています。**OpenPIフレームワーク**で動かす想定のポリシーです。

**`AHegai/pi05_ft_mixed_absolute`** は、`robot.type=so101_follower` 向けに、公式の `lerobot/pi05_base` からファインチューンしたポリシーです。**311エピソード**で学習されており、`lerobot-rollout` コマンドでそのまま動かせる設定例が付属しています。

| 公開ポリシー | ベース | タスク | データ規模 | 実行フレームワーク |
|---|---|---|---|---|
| `felixmayor/pi05_so101_orange_cube` | π0.5 | オレンジキューブのpick&place | 154エピソード / 68.5kフレーム（5000ステップ、loss 0.0315） | OpenPI |
| `AHegai/pi05_ft_mixed_absolute` | `lerobot/pi05_base` | 混合タスク（so101_follower向け） | 311エピソード | `lerobot-rollout` |

これらは、「基盤モデルの土台の上に、SO-101の少数〜中規模のデモを重ねるだけで、実用的なpick&placeポリシーが作れる」ことを示す好例です。まずは自分でゼロからファインチューンする前に、こうした公開ポリシーをダウンロードして動かしてみるのが、π0.5を体感する近道です。

## 19.3 動かすまでの全体像

π0.5でSO-101を動かすには、大きく2つの経路があります。

**経路A：公開ポリシーをそのまま動かす** — `AHegai/pi05_ft_mixed_absolute` のように `lerobot-rollout` で動かせる設定が公開されているポリシーなら、ポリシーを指定して実機（`so101_follower`）で実行するだけで、π0.5の挙動を確認できます。まずはこの経路で「動く状態」を体験するのがおすすめです。

**経路B：自分のタスクでファインチューンする** — 公式の `lerobot/pi05_base` を土台に、自分で集めたSO-101のデモ（第8章の手順で収集し、LeRobotフォーマットに変換したもの）でファインチューンします。`felixmayor` の例のようにOpenPIフレームワークを使う流れ、あるいはLeRobot側のパイプラインでファインチューンする流れがあります。ファインチューン済みのポリシーは、経路Aと同様に実機で実行します。

:::message
📷 **画像プレースホルダー**：π0.5でSO-101を動かす2経路の図。「経路A：公開ポリシーをダウンロード → lerobot-rolloutで実行」と「経路B：pi05_baseを自分のデモでファインチューン → 実行」を並べ、それぞれの入力（公開ポリシー／自分のデータ）と実行コマンドを示す。
:::

いずれの経路でも、実行時は前章のGR00Tと同様、**カメラ画像と言語指示（またはタスク設定）を入力として、π0.5がSO-101の行動を出力する**、という流れになります。

## 19.4 前提とハードウェア要件 — まずACTから

π0.5は強力ですが、**手軽に始められる手法ではありません**。LeRobotの `AGENT_GUIDE.md` でも、π0/π0.5は「SO-101で使えるが、**20GB以上のGPUと、確立したACTベースラインがある前提で検討すべき**」と位置づけられています。これは実務上とても大切な指針です。

つまり、いきなりπ0.5に飛びつくのではなく、

1. **まず第11章のACTで、自分のタスクの土台となるベースラインを作る**（少ないGPU・少ないデモで動き、挙動も理解しやすい）
2. ACTで得られた成功率・失敗パターンを把握したうえで、
3. **さらなる汎化や言語指示対応が必要になったら、20GB以上のGPUを用意してπ0.5のファインチューンを検討する**

という順序が推奨されます。π0.5はACTを置き換えるものというより、ACTで土台を固めた先の「次の一手」と捉えるのが適切です。

| 観点 | ACT（第11章） | π0.5 |
|---|---|---|
| GPU要件 | 控えめ（Colab等で可） | **20GB以上を推奨** |
| 必要データ | 50エピソード程度から | 中規模（数百エピソード規模の例が多い） |
| 言語指示 | 非対応 | 対応（VLA） |
| 汎化性能 | タスク特化 | 高い（基盤モデル） |
| 位置づけ | まず作るベースライン | ベースラインの先の発展 |

:::message
📷 **画像プレースホルダー**：ACTとπ0.5の使い分けを示す図。「ACTでベースライン確立（低GPU・少データ）→ 十分でなければπ0.5へ（20GB+ GPU・言語指示・高汎化）」という段階的な意思決定フローを示す。
:::

## 19.5 まとめ

本章では、VLA基盤モデルの最新版 π0.5 を、SO-101で動かす実践を見ました。要点は、**公式の `lerobot/pi05_base` は汎用ベースモデルであり、SO-101で動かすにはファインチューンされたポリシーが必要**なこと、**そのSO-101向けポリシーは公式リリースではなく、`felixmayor/pi05_so101_orange_cube` や `AHegai/pi05_ft_mixed_absolute` といったコミュニティの成果物として公開されている**こと、そして**π0.5は20GB以上のGPUとACTベースラインを前提に検討すべき「次の一手」**であることです。

まずは公開ポリシーを `lerobot-rollout` で動かして体感し、必要に応じて自分のデータで `pi05_base` をファインチューンしていく——これが、π0.5でSO-101を動かす現実的な道筋です。

## おわりに

本書では、SO-101を組み立て（第1部）、模倣学習でタスクを覚えさせ改善し（第2部）、シミュレーションと基盤モデル、そして最新のVLA（π0.5）まで発展させる（第3部）道のりを、一気通貫でたどってきました。5万円のアームと無料のソフトウェアで、かつては研究室でしかできなかったPhysical AIの実験が、あなたの机の上で回せるようになりました。

そして本章で見たように、あなた自身がファインチューンして公開したポリシーが、次に誰かの出発点になります。ここから先は、あなたのタスクを設計し、データを集め、モデルを学習させ、コミュニティにデータセットやモデルを公開していく番です。次のブレークスルーを生むのは、あなたかもしれません。

## 付録A　トラブルシューティング集約

本書の各章で触れたトラブルと対処を、フェーズ別に一覧化します。詳細は各章を参照してください。

### A.1 環境構築（第5章）

| 症状 | 対処 |
|---|---|
| `pip install` が wheel ビルドで失敗 | `xcode-select --install`（Xcode CLT導入） |
| `conda: command not found` | `conda init zsh` してターミナル再起動 |
| MPS available: False | macOS 12.3未満かIntel Mac。学習はColabへ |
| Hugging Faceログイン失敗 | トークンを **Write** 権限で再発行、`echo $HF_TOKEN` を確認 |
| Rerunの属性エラー | `pip install --upgrade rerun-sdk` |

### A.2 ハードウェア・キャリブレーション（第6〜7章）

| 症状 | 対処 |
|---|---|
| モーターが検出されない | ケーブル・電源・ポート名を確認、別USBポート |
| ID設定ができない | モーターは1台ずつ接続、ファームウェア更新、ボーレート1000000 |
| リーダーとフォロワーがずれる | ゼロ位置を慎重に再設定、モーターのラベルを再確認 |
| ファームウェア更新が失敗 | モーター再接続、ファイル再ダウンロード、安定した電源 |
| テレオペが震える／脱力する | PIDゲイン調整／電源供給を確認 |

### A.3 データ収集・学習・評価（第8〜13章）

| 症状 | 対処 |
|---|---|
| シリアルポートエラー | `lerobot-find-port` で再確認 |
| カメラが開けない | camera_index=0,1,2 を試す、他アプリの占有を確認 |
| 学習が進まない（loss横ばい） | データの多様性を増やす |
| GPU Out of Memory | `--batch_size` を4〜8に下げる |
| wandb容量不足 | 古いアーティファクトを削除（無料枠5GB） |
| 実機で動かない | 最低6万ステップは学習する |
| 動作が震える | Temporal Ensembleを有効化、chunk_size調整 |
| 特定状況だけ失敗 | その状況のデータを重点的に追加（第13章） |

### A.4 シミュレーション（第14〜17章）

| 症状 | 対処 |
|---|---|
| Isaac SimのFPSが低い | 不要な拡張を無効化、レンダリング設定を軽量化 |
| Simで成功・実機で失敗 | 物理パラメータを実機に合わせる、Domain Randomization、センサーノイズ |
| ROS2で余計なトピック混入 | `ROS_DOMAIN_ID` で独自ドメインに分離 |
| Cosmos/VLAでGPUメモリ不足 | バッチサイズ・並列環境数を減らす、Mixed Precision |

## 付録B　用語集

- **LeRobot**：Hugging Faceが主導するオープンソースのロボティクス学習フレームワーク。データ収集・学習・評価を統一的に扱う。
- **SO-101**：LeRobotの定番ハードウェアであるオープンソースのロボットアーム。リーダーアームとフォロワーアームの2台構成。
- **テレオペレーション**：人間がリーダーアームを動かすと、フォロワーアームが追従する遠隔操作。データ収集の基礎。
- **エピソード / フレーム / ステップ**：エピソードはタスク1回分の記録、フレーム/ステップはその中の各サンプル点。
- **模倣学習（Imitation Learning）**：人間のデモから行動方策を学ぶ手法。
- **ACT（Action Chunking Transformer）**：複数ステップ先の行動をまとめて予測する模倣学習手法。SO-101の第一選択。
- **Action Chunking**：1ステップずつではなく、未来の複数ステップの行動をまとめて予測すること。
- **Temporal Ensemble**：複数時刻の予測を加重平均して動作を滑らかにする手法。
- **Diffusion Policy**：拡散モデルで行動を生成する手法。マルチモーダルに強いが推論が遅い。
- **強化学習（RL）**：報酬を最大化するよう自律的に学習する手法。SAC、RLPD、PPOなど。
- **HIL-SERL**：人間の介入を活用する強化学習。報酬分類器＋オンラインRL＋人間介入の3フェーズ。
- **VLA（Vision-Language-Action）**：視覚・言語・行動を統合する基盤モデル。π0、GR00T、SmolVLA、Hi Robotなど。
- **π0（Pi Zero）／GR00T N1.5**：代表的なVLA基盤モデル。少数デモでファインチューニングできる。
- **Isaac Sim / Isaac Lab**：NVIDIAの物理シミュレーター／その上のロボット学習フレームワーク。
- **NVIDIA Cosmos**：生成AIによるWorld Foundation Model。Predict/Transfer/Reasonの3モデル。
- **Genesis**：Macでも動く軽量な物理シミュレーター。並列強化学習に対応。
- **World Model（世界モデル）**：世界の動き方を予測する内部モデル。
- **Sim-to-Real / Reality Gap**：シミュレーションで学んだものを実機に移すこと／その間のギャップ。
- **Domain Randomization**：学習時に物理・視覚パラメータをランダム化し、実機のばらつきに頑健にする手法。
- **クロスエンボディメント**：複数の異なるロボットにまたがってスキル・ポリシーを共有・転移すること。
- **URDF / USD**：ロボットの物理モデルを記述するフォーマット（URDF）／Omniverseのシーン記述（OpenUSD）。
