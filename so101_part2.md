# 第8章　データセット作成の実践
---

第1部で、SO-101を組み立ててテレオペで動かせるようになりました。本章では、模倣学習の要である**データセット作成**——タスクを人手で実演し、カメラ画像とアームの動きを記録する——を実践します。学習アルゴリズム（ACT）の詳細は第11章、学習済みモデルの評価は第13章で扱うため、ここでは**良質なデータをどう集めるか**に集中します。

## 8.1 模倣学習とデータ収集の位置づけ

模倣学習は、人間のデモンストレーションからタスクの方策（Policy）を獲得する手法で、①データ収集（人がリーダーアームで実演）→②学習（モデルが行動パターンを習得）→③実行（自律動作）→④評価（成功率測定）→⑤改善（データ追加・再学習）というループを回します。プログラミング不要でデモするだけ、環境変化に柔軟、画像から直接行動を生成できる——これが模倣学習の利点です。

そしてこのループのなかで**最も成果を左右するのがデータ収集**です。どんなに優れたアルゴリズムでも、質の悪いデータからは良いモデルは生まれません。

## 8.2 事前準備

必要なアカウントは、データセット/モデルの保存・共有に使う **Hugging Face**（Write権限トークン）、学習進捗を監視する **Weights & Biases（wandb）**（無料枠で十分）、そして学習用GPUとしての **Google Colab Pro**（月額約1,200円、A100/L4が使える）です。環境は第5章で構築済みのものを使い、`huggingface-cli login` と `wandb login` を済ませておきます。

```bash
huggingface-cli login   # Write権限トークンを入力
HF_USER=$(huggingface-cli whoami | head -n 1)   # ユーザー名を取得
wandb login
```

**ロボット・カメラの配置**は、フォロワーアームを作業領域の中央に、リーダーアームはカメラに映らない位置に置き、カメラは前方斜め上（45〜60度）から作業領域全体を俯瞰させます。作業領域全体が画角に収まり、対象物とゴールが明確に見え、影が最小になるよう照明を調整し、リーダーアームが映り込まないようにします。

![画像](https://static.zenn.studio/user-upload/dc4d426a0f2a-20260106.jpeg)

:::message
📷 **画像プレースホルダー**：データ収集時のカメラ・アーム配置を上から見た「レイアウト図」。フォロワーアーム（作業領域中央）、俯瞰カメラ（45〜60度）、対象物とゴール地点、リーダーアーム（画角外）の位置関係を寸法感つきで示す。
:::

## 8.3 データ収集の実践

本章では定番の **Pick and Place**（ブロックを掴んで目標エリアに置く）タスクを例にします。シンプルですが、位置や向きの変化への対応が求められる良い題材です。

![画像](https://static.zenn.studio/user-upload/ff92b8ebe206-20251223.png)

データ収集は `record` コマンドで行います（引数はLeRobotのバージョンで変わる場合があります）。

```bash
python -m lerobot.record \
  --robot.type=so101_follower --robot.port=<フォロワーのポート> \
  --teleop.type=so101_leader --teleop.port=<リーダーのポート> \
  --dataset.repo_id=${HF_USER}/so101_pick_cube \
  --dataset.single_task="Grasp a lego block and put it in the bin." \
  --dataset.num_episodes=50 \
  --dataset.fps=30 \
  --dataset.push_to_hub=true
```

主なパラメータは、記録フレームレート（`fps=30`）、タスク説明文（`single_task`）、Hugging Faceリポジトリ名（`repo_id`）、収集エピソード数（`num_episodes`）、そしてHubへの自動アップロード（`push_to_hub`）です。ウォームアップ時間・エピソード最大時間・リセット時間なども指定できます。

**収集の流れ**は、①スクリプト起動（ロボット・カメラ初期化）→②「Recording episode 1/50」の表示とともにリーダーアームで実演（フォロワーが追従し、カメラ映像とロボット状態を記録）→③キーボードで操作（右矢印＝次へ、左矢印＝やり直し、ESC＝終了）→④リセット時間で対象物の位置を変更→⑤全エピソード完了まで繰り返し→⑥Hubへ自動アップロード、というものです。

![画像](https://static.zenn.studio/user-upload/652ccc611fc1-20260106.gif)

## 8.4 データ収集のコツ

**操作**：リーダーアームはゆっくり動かし、前腕を反対の手で支え、急な動きを避け、グリッパーの開閉を明確に行います。

**データの多様性**：最初は10エピソード単位で集め、対象物の位置・向き・ゴール位置を少しずつ変えます。失敗したエピソードも多様性のために一定数含めると、モデルが復帰行動を学べます。

**品質管理**：10エピソードごとにビジュアライザで確認し、画角から物体が切れていないか、人間でも実行可能なタスクか、を点検し、明らかな失敗は左矢印でやり直します。

**目標量**：まず50エピソード、本格的には100〜200エピソード。段階的にバリエーションを増やすのがコツです（Phase 1：固定位置10 → Phase 2：位置変更20 → Phase 3：向きも変更30 → Phase 4：ゴール位置も変更50 → 評価して追加）。「何ステップ・何エピソード集めるべきか」という定量的な設計は、次章（第9章）・第10章で詳しく扱います。

## 8.5 データセットの確認

収集後は、必ず中身を確認します。ローカルまたはHugging FaceのWebビジュアライザで、各エピソードの映像と記録を閲覧できます。

```bash
python -m lerobot.scripts.visualize_dataset_html --repo-id ${HF_USER}/so101_pick_cube
```

![画像](https://static.zenn.studio/user-upload/02ae423630a2-20251223.png)

収集したデータは実機で**リプレイ**して再現性を確認することもできます（同じエピソードなら常に同じ動きになります）。データは `~/.cache/huggingface/lerobot/${HF_USER}/so101_pick_cube` に保存され、Hubにもアップロードされます。

![画像](https://static.zenn.studio/user-upload/2240cf099865-20251223.png)

## 8.6 学習と評価への橋渡し

データが揃ったら、Google Colab等のGPUでACTモデルを学習します（詳細な手順・ハイパーパラメータは第11章）。最小形は次の通りです。

```bash
python -m lerobot.scripts.train \
  --dataset.repo_id=${HF_USER}/so101_pick_cube \
  --policy.type=act \
  --output_dir=outputs/train/pick_cube \
  --wandb.enable=true --steps=100000
```

学習はwandbのダッシュボードで監視し、`train/loss` が順調に下がっていれば正常です（横ばい・上昇は過学習やデータ不足のサイン）。学習時間の目安は50エピソードでA100約90分、RTX 3060で約4〜6時間。推奨学習量は標準10万ステップ、高品質なら15〜20万ステップです。

![画像](https://static.zenn.studio/user-upload/d1a8e0a36d77-20251225.png)

学習が終わったら、学習済みモデルを実機で動かして成功率を測ります（詳細は第13章）。この「評価→失敗分析→データ追加→再学習」のループを回すのが上達の近道です。

![画像](https://static.zenn.studio/user-upload/4c25ed55ab2f-20260106.gif)

## 8.7 トラブルシューティング

| 問題 | 原因 | 解決策 |
|---|---|---|
| シリアルポートエラー | ポート設定ミス | `lerobot-find-port` でポートを再確認 |
| カメラが開けない | デバイスID違い | camera_index=0,1,2 を試す |
| 学習が進まない（loss横ばい） | データの多様性不足 | 位置・向きのバリエーションを追加 |
| GPU Out of Memory | バッチサイズ過大 | `--batch_size=4` などに下げる |
| wandb容量不足 | 無料枠5GB到達 | 古いアーティファクトを削除 |
| 実機で動かない | 学習不足 | 最低6万ステップは学習する |

## 8.8 まとめ

本章では、Pick and Placeを例にデータセット作成の実践を学びました。要点は、**カメラ・アームの配置を固定し照明を一定に保つ**こと、**位置・向き・ゴールにバリエーションを持たせる**こと、**10エピソードごとに品質確認する**こと、そして**段階的に量を増やす**ことです。推奨ワークフローは「環境準備 → 小規模収集（10）→ テスト学習 → 実機評価 → 本格収集（50）→ 本学習 → 最終評価 → 追加収集」の反復です。

次章では、「そもそも何ステップ・何エピソード集めればよいのか」というデータ設計の勘所を、主要データセットの統計から読み解きます。

# 第9章　主要データセットのエピソード長とステップ数
---

前章でデータ収集を実践しましたが、そこで必ず直面する疑問が「**1エピソードはどのくらいの長さにすべきか？何エピソード集めればよいのか？**」です。本章では、著名なロボット学習データセットの統計からその勘所を読み解きます。

## 9.1 エピソードとステップの基本

**エピソード**は、タスクの開始から目標達成（または失敗）までの一連の動作＝1回のデモンストレーションです。**ステップ**は、その中の個々の制御サイクル（各ステップでロボットは観測を受け取り行動を出力）です。両者は制御周波数で結ばれます。

```
実時間 = ステップ数 ÷ 制御周波数
例： 38ステップ ÷ 5Hz = 7.6秒 ／ 100ステップ ÷ 10Hz = 10秒
```

## 9.2 主要データセットの統計

代表的なデータセットを比較すると、エピソード長には大きな幅があることが分かります。

| データセット | 平均ステップ数 | 制御周波数 | 平均実時間 | タスク複雑度 |
|---|---|---|---|---|
| BridgeData V2（Berkeley/Stanford, 2023） | 38 | 5Hz | 約7.6秒 | 低〜中 |
| RH20T（上海交通大学, 2024） | — | 多様 | 10〜100秒 | 高 |
| Open X-Embodiment（2023） | 約120 | 3〜10Hz | 12〜40秒 | 中〜高 |
| RT-1（Google, 2022） | — | — | 短め | 中 |
| ALOHA（Stanford, 2023） | 50〜200 | 10〜50Hz | 5〜20秒 | 中〜高 |

**BridgeData V2** は60,096エピソードを5Hz・平均38ステップという短く効率的な設計で集めており、シンプルなタスクに特化しています。**RH20T** は147タスク・RGB-D/力覚/音声のマルチモーダルで、10〜100秒の長く複雑なエピソードを含みます。**Open X-Embodiment** は100万超のエピソードを22種類のロボットから集約した標準フォーマット（RLDS）で、クロスエンボディメント学習を志向します。**ALOHA** は両腕ロボットで、精密な接触作業のため高周波数（最大50Hz）を使います。

:::message
📷 **画像プレースホルダー**：主要データセットの「平均エピソード長（実時間）」を横棒グラフで比較した図。BridgeData V2（短）からRH20T（長）まで並べ、タスク複雑度を色で示す。「短く効率的」という潮流が一目でわかる図。
:::

## 9.3 タスク複雑度別の推奨値

タスクの複雑さに応じて、エピソード長の目安は次のように整理できます。

| タスクタイプ | ステップ数 | 実時間(10Hz) | 例 |
|---|---|---|---|
| 超シンプル | 20〜30 | 2〜3秒 | ボタン押し、単純動作 |
| シンプル | 30〜50 | 3〜5秒 | Pick-and-place、Push |
| 標準 | 50〜100 | 5〜10秒 | 引き出し開閉 |
| 中程度 | 100〜150 | 10〜15秒 | 2段階タスク |
| 複雑 | 150〜250 | 15〜25秒 | 複数段階の操作 |
| 超複雑 | 250+ | 25秒+ | **分解を推奨** |

**シンプルタスク**は5〜10Hz・30〜50ステップが目安で、データ収集が速く学習も容易です（BridgeData V2の38ステップが好例）。**中程度タスク**は10Hz・80〜120ステップで、サブゴールを明確にし、失敗時は早期終了させるのがポイントです。**複雑タスク**は準備・実行・検証といった段階に分けて記録します。**超複雑タスク（300ステップ超）**は学習が困難なため、階層的分解（長いタスクを50〜100ステップのサブタスクに分ける）やカリキュラム学習（易→難へ段階的に）を推奨します。

## 9.4 制御周波数の選択

制御周波数は、データサイズと制御の精度のトレードオフです。

- **低周波数（5Hz）**：データサイズが小さく計算負荷が低い。ゆっくりした操作や静的なシーン向き（BridgeData V2）。
- **中周波数（10〜20Hz）**：バランスが良く、多くのタスクで十分。標準的な選択。
- **高周波数（50Hz+）**：精密な制御が可能だがデータサイズと計算コストが大きい。接触を伴う精密作業向き（ALOHA）。

SO-101での標準は30Hz程度ですが、タスクに応じて調整します。

## 9.5 データ収集効率とタスク分解の判断

データ収集効率は「1時間あたりに集められる成功エピソード数」で考えます。1エピソードあたりの時間は「セットアップ時間＋実行時間」で、成功率を掛けたものが有効な収集速度です。たとえば38ステップ（BridgeスタイルのPick-and-Place）・成功率85%なら、約90エピソード/時間 × 0.85 ≈ 77成功エピソード/時間になります。**1時間に50〜100エピソードを目標**にするとよいでしょう。

タスクの実行時間が30秒を超える、あるいは成功率が30%を下回る場合は、**タスク分解を検討**します。積み上げタスクなら「1個積む（50エピソード）→2個積む（100エピソード）→3個積む（100エピソード）」と段階的に集めると、転移学習効果で全体の収集・学習効率が上がります。

:::message
📷 **画像プレースホルダー**：タスク複雑度（横軸）に対する「推奨エピソード長」と「分解すべきライン」を示した図。250ステップ超で「階層的分解を推奨」の領域に入る境界を明示する。
:::

## 9.6 データセット設計のベストプラクティス

**エピソード長の一貫性**：同種タスクではステップ数のばらつき（標準偏差）を小さく保ちます。平均50でも標準偏差30のようにばらつきが大きいと、学習が不安定になります。

**早期終了の実装**：成功したら終了、タイムアウトしたら終了、回復不能な失敗をしたら終了、という条件を設けると、無駄なデータを減らせます。

**メタデータの記録**：各エピソードに、総ステップ数・実時間・制御周波数・成功/失敗・タスク名・対象物・環境などを記録しておくと、後の分析やフィルタリングに役立ちます。

## 9.7 トラブルシューティング

| 症状 | 対処 |
|---|---|
| エピソードが長すぎる（収集が遅い・学習が収束しない） | タスクを分解、早期終了を追加、制御周波数を下げる |
| エピソードが短すぎる（タスクが完了しない） | ステップ数を増やす、周波数を上げる、動作を丁寧に |
| 成功率が低い（収集効率が悪い） | 初期条件を制約、タスクを簡単に、環境を整える |

## 9.8 まとめ

エピソード長の選択は、データセット設計で最も重要な決定の一つです。**短く、シンプルに、効率的に**——これが成功の鍵です。BridgeData V2の38ステップは、長年の研究と実践から得られたベンチマークであり、あなたのプロジェクトでもこれを出発点に、タスクに応じて調整するとよいでしょう。100ステップを超えるタスクは分解を検討し、1時間に50〜100エピソードの収集効率を目標にします。

次章では、実際に公開されているLeRobotデータセットのエコシステムを探索し、既存データの活用方法を見ていきます。

# 第10章　LeRobotデータセットエコシステムの探索
---

SO-101でロボット学習を始めると、「他の人たちはどんなタスクに取り組んでいるのか？」という疑問が湧きます。本章では、Hugging Face上に公開されているLeRobotデータセットのエコシステムを探索します。既存データセットは、自分のプロジェクトのアイデア源にも、ベンチマークにも、転移学習のソースにもなります。

![画像](https://static.zenn.studio/user-upload/ae3aef80afa6-20260109.png)

データは [Dataset Visualizer](https://huggingface.co/spaces/lerobot/visualize_dataset) で視覚的に確認できます。

## 10.1 データセットの基本構造

LeRobotデータセットは、**エピソード**（タスク開始から終了までの一連の動作記録）と、その中の各サンプル点である**フレーム**からなります。通常30fpsで記録され、各フレームには関節角度（`observation.state`）、アクション指令（`action`）、カメラ画像（`observation.images`）、タイムスタンプが含まれます。

規模は簡単に計算できます。たとえば `lerobot/svla_so100_sorting`（52エピソード、35,713フレーム、30fps）なら、総記録時間は `35,713 ÷ 30 ≈ 1,190秒 ≈ 約19.8分`、1エピソードあたり `35,713 ÷ 52 ≈ 687フレーム ≈ 約23秒` となります。

## 10.2 SO-101/SO-100用データセット

SO-101/SO-100向けには、難易度が段階的に上がるタスクが揃っています。

| データセット | エピソード | フレーム | 記録時間 | 難易度 |
|---|---|---|---|---|
| `svla_so101_pickplace` | 50 | 11,939 | 約6.6分 | 基本（Pick and Place） |
| `svla_so100_pickplace` | 50 | 19,631 | 約10.9分 | 基本 |
| `svla_so100_stacking` | 56 | 22,956 | 約12.8分 | 精密（積み上げ） |
| `svla_so100_sorting` | 52 | 35,713 | 約19.8分 | 複合（仕分け・最大規模） |

**Pick and Place** は最も基礎的で、初めてのデータセット作成の参考になります。**Stacking（積み上げ）** は精密な空間認識と接触制御、バランス維持が必要です。**Sorting（仕分け）** は複数物体の認識と分類を伴い、実用的な家事タスクに近い複雑さを持ちます（v3.0の最新フォーマット）。

![画像](https://static.zenn.studio/user-upload/28d136193036-20260108.png)

自分のプロジェクトに合わせて選ぶポイントは、**タスクの複雑さ**（Pick and Place → Stacking → Sorting の順に難化）、**規模**（フレーム数が多いほど長時間の操作記録）、**フォーマットバージョン**（v3.0はストリーミング等の新機能に対応）です。

## 10.3 ALOHA双腕ロボットのタスク

SO-101が単腕なのに対し、ALOHAは両腕を持ち、より人間に近い操作ができます。

- **`aloha_mobile_cabinet`**（85エピソード、127,500フレーム、約42.5分、4カメラ・50fps）：移動しながらキャビネットを操作する Navigation + Manipulation の複合タスク。SO-101データの10倍以上の規模。

# 第11章　ACT（Action Chunking Transformer）による模倣学習
---

本章では、SO-101で最も効果的な模倣学習手法の一つ **ACT（Action Chunking with Transformers）** を解説します。ACTは、従来の模倣学習が抱えていた課題を解決し、少ないデモから高精度な方策を学習できる手法です（Zhao et al., 2023）。

## 11.1 模倣学習の根本的な課題

ACTが解こうとする課題は2つあります。

**課題1：人間のデモの一貫性のなさ** — 人間が同じタスクを繰り返しても、毎回まったく同じ動作をするわけではありません。ブロックを掴む動作でも「上から接近して一時停止して掴む」「横から接近してすぐ掴む」など、ゴールは同じでも経路が異なります。学習モデルがこれらを単純に平均しようとすると、どのパターンも正しく再現できない中途半端な方策になってしまいます。

**課題2：非マルコフ性** — 「次の行動は現在の状態だけで決まる」というマルコフ性を仮定すると、困った事態が起きます。たとえば「アームが物体の上で停止している（グリッパー開）」状態は、位置調整中の停止（次は下降して掴む）かもしれないし、別の場面かもしれません。現在の状態だけでは、次に何をすべきか区別できないのです。極端な場合、「停止状態を見て停止と判断し、また停止状態になり…」という無限ループに陥ります。

:::message
📷 **画像プレースホルダー**：非マルコフ性の問題図。「同じ状態（アームが物体上で停止）」から「下降して掴む」と「目標へ移動する」という異なる正解に分岐する様子を示し、現在状態だけでは区別できないことを図解する。
:::

## 11.2 ACTの2つの鍵

ACTはこの2つの課題を、**Transformer** と **Action Chunking** で解決します。

**Transformerによる文脈の考慮** — TransformerのSelf-Attentionにより、現在の状態だけでなく**過去の状態・行動の系列全体**を考慮して行動を決めます。「どういう経緯でこの状態に来たか」が分かるため、非マルコフ性の問題を克服できます。また、複数の解き方が存在することを前提とした学習ができるため、デモのばらつきにも対応できます。

**Action Chunking（行動のまとめ予測）** — 従来は「時刻tの状態→時刻tの行動」を1ステップずつ予測していましたが、ACTは「時刻tの状態→時刻t〜t+kの行動系列（チャンク）をまとめて予測」します。

```
従来：  action_t     = policy(state_t)              # 1ステップ
ACT：   action_chunk = policy(state_t)              # k ステップ分をまとめて
        # action_chunk = [action_t, action_{t+1}, …, action_{t+k}]
```

これにより、①**動作が滑らかになる**（数ステップ先を見据えるため関節角度が急変しない）、②**誤差の累積が抑えられる**（軌道全体を最適化できる）、③**実行効率が上がる**（一度の推論で複数ステップ分を取得）というメリットが生まれます。

## 11.3 ACTのアーキテクチャ

ACTは3つのコンポーネントからなります。

1. **エンコーダー（CNN または ViT）**：カメラ画像から特徴量を抽出
2. **Transformer Encoder-Decoder**：状態系列をエンコードし、Temporal attentionで時系列の文脈を理解
3. **Action Chunking機構**：複数ステップ先の行動系列をまとめて予測

内部的には Conditional VAE（条件付き変分オートエンコーダ）を用いており、デモの多峰性（複数の解き方）を潜在変数で表現します。

:::message
📷 **画像プレースホルダー**：ACTのアーキテクチャ図。「カメラ画像＋関節角度 → 画像エンコーダー（CNN/ViT）→ Transformer Encoder-Decoder → 未来kステップの行動チャンク出力」という流れを、Conditional VAEの潜在変数も含めて示す。
:::

## 11.4 Temporal Ensemble による滑らかな実行

Action Chunkingには弱点もあります。チャンクの終わり際は予測精度が落ちがちで、チャンクの切れ目で動作がカクつくことがあります。これを解決するのが **Temporal Ensemble** です。

毎ステップ新しいチャンクを予測し、**過去に予測した複数のチャンクのうち、その時刻に該当する予測を加重平均**します。時刻tの行動は、「t−2で予測したチャンクのt番目」「t−1で予測したチャンクのt番目」「tで予測したチャンクの先頭」を重み付き平均する、というイメージです。これにより、チャンクの切れ目でも滑らかで安定した動作が実現します。

:::message
📷 **画像プレースホルダー**：Temporal Ensembleの概念図。複数時刻で予測されたチャンク（重なり合う横棒）が、ある時刻で加重平均される様子を示す。切れ目のカクつきが平滑化されることが視覚的に分かる図。
:::

## 11.5 SO-101での実装

LeRobotではACTが標準サポートされており、データセットを指定して学習するだけです（第8章の手順）。

```bash
python -m lerobot.scripts.train \
  --dataset.repo_id=${HF_USER}/so101_pick_cube \
  --policy.type=act \
  --policy.chunk_size=10 \
  --output_dir=outputs/train/pick_cube \
  --wandb.enable=true --steps=100000
```

推論時は、`chunk_size` 分の行動をまとめて取得し、Temporal Ensembleで平滑化しながら30Hzで実行します。学習・推論の具体的なコマンドは第8章・第13章を参照してください。

## 11.6 性能向上のテクニック

**データ拡張** — 画像の色調変化（ColorJitter）、ぼかし、わずかな平行移動・回転を加えると、照明や配置の変化に強くなります。行動空間は平均・標準偏差で正規化しておきます。

**Chunk sizeの調整** — チャンクが小さすぎると長期的な計画能力が不足し、大きすぎると予測の不確実性と計算コストが増えます。SO-101では10〜15が最適な場合が多く、実験では chunk_size=10 で成功率が最も高かった、という報告があります。

| Chunk size | 成功率（例） |
|---|---|
| 5 | 75% |
| **10** | **92%** ← 最適 |
| 15 | 88% |
| 20 | 80% |
| 30 | 70% |

:::message
📷 **画像プレースホルダー**：Chunk sizeに対する成功率の折れ線グラフ（5→92%のピークが10、以降低下）。「小さすぎず大きすぎず」の最適点が一目でわかる図。
:::

**マルチモーダル学習** — 複数カメラ（俯瞰＋手首）の特徴を融合すると、単一視点では見えない情報を補え、精度が向上します。

## 11.7 評価とデバッグ

学習済みポリシーは、初期姿勢にリセット→自律実行→成功/失敗を記録、を繰り返して成功率を測ります（詳細は第13章）。デバッグには、①各関節の予測行動系列をプロットして急変がないか確認する、②Self-Attentionの重みをヒートマップ表示してモデルがどこに注目しているか見る、③同じ観測に対する予測のばらつき（分散）を測る（低ければ安定、高ければ不安定）、といった手法が有効です。

| 症状 | 原因 | 対処 |
|---|---|---|
| 訓練損失が下がらない | 学習率が不適切 | 1e-5〜1e-4で調整 |
| 評価時に動作が不安定 | Chunk sizeが大きすぎ | 10前後に調整 |
| ロボットが振動する | Temporal Ensembleの未活用 | 予測の平滑化を有効に |
| メモリ不足 | Batch sizeが大きすぎ | 4〜8に削減 |
| 過学習 | データ拡張不足 | Augmentationを追加 |

## 11.8 ACTとDiffusion Policyの比較

ACTと並んで注目される **Diffusion Policy** との比較を示します。

| 特徴 | ACT | Diffusion Policy |
|---|---|---|
| 推論速度 | 高速（リアルタイム可） | 遅い（反復サンプリング必要） |
| 学習安定性 | 高い | やや低い |
| マルチモーダル性 | VAEで対応 | 自然に対応 |
| 実装の複雑さ | 中程度 | 高い |
| SO-101での推奨 | ◎ | △ |

リアルタイム制御が求められるSO-101では、推論速度と学習安定性に優れるACTが第一候補になります。

## 11.9 まとめ

本章では、ACTの原理と実装を解説しました。要点は、**Transformerのattentionで過去の系列を考慮し非マルコフ性を克服する**こと、**Action Chunkingで複数ステップ先をまとめて予測し滑らかで一貫性のある動作を得る**こと、**Temporal Ensembleでチャンクの切れ目を平滑化する**こと、そして**Chunk sizeは10前後が最適な場合が多い**ことです。リアルタイム制御と高い成功率を両立できるACTは、SO-101での模倣学習の第一選択です。

# 第12章　ロボット学習の主要手法とHIL-SERLの実践
---

前章のACTは模倣学習の代表格ですが、ロボット学習には他にも多くの手法があります。本章では主要手法を俯瞰し、模倣学習と強化学習を橋渡しする **HIL-SERL（Human-in-the-Loop強化学習）** を中心に、手法の選び方を整理します。

## 12.1 ロボット学習手法の全体像

ロボット学習は大きく3つに分類されます。

- **模倣学習（Imitation Learning）**：人間のデモから直接学習。報酬設計が不要で安全だが、デモの質と量に依存する。
- **強化学習（Reinforcement Learning）**：報酬を最大化するよう自律的に学習。人間が思いつかない解法を発見できるが、サンプル効率が課題。
- **ハイブリッド**：両者の利点を組み合わせた手法で、現在の主流。HIL-SERLはこの代表例です。

:::message
📷 **画像プレースホルダー**：ロボット学習手法のマップ図。横軸を「模倣学習 ⇔ 強化学習」、縦軸を「データ効率／汎化性能」などにして、ACT・Diffusion Policy・HIL-SERL・DAgger・VLAを配置した俯瞰図。各手法の立ち位置が一目でわかる図。
:::

## 12.2 主要な模倣学習手法

**ACT（Action Chunking with Transformers）**（第11章）は、約80Mパラメータと軽量で、50個程度のデモから高い成功率（両手操作や精密接触タスクで80〜90%）を達成します。誤差の累積を抑え時間的一貫性に優れる一方、リアルタイムのフィードバックには弱い（閉ループ制御が苦手）という特性があります。

**Diffusion Policy** は、画像生成の拡散モデルをロボット制御に応用し、ノイズ除去プロセスでアクションを生成します。複数の解決方法がある**マルチモーダルな行動分布**を自然に扱え、12のベンチマークで平均46.9%の性能改善を示しました。一方で推論が遅く、視覚サーボのような閉ループ反応型タスクでは性能が落ちます。

## 12.3 強化学習ベースの手法

**SAC（Soft Actor-Critic）** は、エントロピー正則化により探索と活用のバランスを自動調整するoff-policy手法です。**RLPD（Reinforcement Learning with Prior Data）** は、オフラインのデモデータとオンライン探索データを両方バッファに保存し、均等にサンプリングすることでサンプル効率を高めます。このRLPDが、次のHIL-SERLの学習部分の基盤になります。

## 12.4 HIL-SERL：Human-in-the-Loop強化学習

**HIL-SERL** はUC BerkeleyのRAILが開発した人間参加型の強化学習で、精密で器用な操作タスクを学習できます。3つのフェーズで構成されます。

**Phase 1：報酬分類器の訓練** — 成功例・失敗例の画像を集め、ResNet18などをバックボーンにした二値分類器を訓練します。この分類器が「タスクが成功したか」を画像から判定し、強化学習の報酬信号（成功=1、失敗=0）を自動生成します。

**Phase 2：初期デモの収集** — SpaceMouseやゲームパッドで20〜30個の成功デモを集め、デモバッファに追加します。

**Phase 3：オンライン強化学習＋人間介入** — Actor（環境と相互作用）とLearner（ポリシー更新）が並行し、ポリシーが失敗しそうな時に**人間が介入して正しい行動を示します**。介入前後のデータはRLバッファに入り、デモバッファと混ぜて（RLPDで）学習されます。

:::message
📷 **画像プレースホルダー**：HIL-SERLの3フェーズ図。「報酬分類器の訓練 → 初期デモ収集 → オンラインRL＋人間介入」の流れと、Actor-Learnerアーキテクチャ（デモバッファ／RLバッファ／人間の介入）を1枚に示す。
:::

**人間介入のコツ**は、初期は頻繁に介入し、成功率が上がるにつれて徐々に減らすことです。介入のタイミングは、①失敗を防ぐ（明らかに失敗しそうな時）、②新しい状態を示す（訓練データに少ない状況での正しい行動）、③ローカルミニマムからの脱出を支援する時、です。成功率とエピソード数に応じて介入確率を下げていくと、効率的に学習が進みます。

LeRobotにはHIL-SERLが実装されており、環境・報酬分類器・介入デバイス（ゲームパッド等）・目標成功率を指定して訓練できます。

## 12.5 DAgger系とVLA系

**CR-DAgger（Compliant Residual DAgger）** は、力覚制御を使って実行中のポリシーを中断せずに優しく修正でき、50個の修正エピソードで基本ポリシーの成功率を50%以上改善します。**Diff-DAgger** は、Diffusion Policyで不確実性を推定し介入タイミングを決めるため、マルチモーダルな分岐点で誤って介入しません。

**Hi Robot** は階層的なVLA（Vision-Language-Action）モデルで、高レベルの推論（複雑な指示を原子的コマンドに分解）と低レベルの実行（コマンドをアクションに変換）の両方にVLMを使います。「ベジタリアンサンドイッチを作って」のような複雑な指示を処理でき、GPT-4oより40%以上高い指示精度を示します（VLA基盤モデルは第18章で詳しく扱います）。

## 12.6 Cosmosとの統合（Sim-to-Real）

NVIDIA CosmosのWorld Modelで生成した物理シミュレーションデータを、実データと混ぜて学習に使うことで、ロバスト性を高められます（第15章）。多様な初期条件・外乱でシミュレーションデータを生成し、実データと3:7程度の比率で統合するのが一例です。Sim-to-Realギャップは、物理パラメータ（摩擦・質量）や照明・カメラノイズをランダム化する**Domain Randomization**、シミュレーション画像を実画像風に変換する**Domain Adaptation**で軽減します。

## 12.7 手法の比較と選択ガイド

| 手法 | データ効率 | 訓練時間 | 成功率 | 汎化性能 | リアルタイム性 |
|---|---|---|---|---|---|
| ACT | ★★★★☆ | ★★★★★ | ★★★★☆ | ★★★☆☆ | ★★★★★ |
| Diffusion Policy | ★★★☆☆ | ★★★☆☆ | ★★★★★ | ★★★★☆ | ★★☆☆☆ |
| HIL-SERL | ★★★★★ | ★★★☆☆ | ★★★★★ | ★★★★★ | ★★★★☆ |
| DAgger | ★★★☆☆ | ★★★★☆ | ★★★☆☆ | ★★★☆☆ | ★★★★☆ |
| Hi Robot（VLA） | ★★☆☆☆ | ★★☆☆☆ | ★★★★☆ | ★★★★★ | ★★★☆☆ |

タスク別には、**精密操作（組み立て・挿入）**はHIL-SERL、**多様な解法があるタスク（片付け・整理）**はDiffusion Policy、**長期複雑タスク（料理・清掃）**はHi Robot、**データが限られる場合**はACT（50デモから可能）が第一候補です。

**SO-101での実践推奨**は段階的なアプローチです。まず**ACTで基本ポリシーを学習**し、成功率が不十分（80%未満）なら**HIL-SERLでファインチューニング**（ACTの成功/失敗ロールアウトから報酬分類器を作り、ゲームパッド介入で改善）、さらにロバスト性が必要なら**Cosmosの合成データで補強**する、という流れです。

## 12.8 トラブルシューティング

| 症状 | 原因 | 対処 |
|---|---|---|
| ポリシーが震える | Temporal Ensemble不足、高周波ノイズ | ACTのensembling重みを調整、Diffusionのステップ数を増やす |
| ポリシーが保守的すぎる | KL損失・エントロピー正則化が強すぎ | ACTの`kl_weight`を下げる、SACの温度を下げる |
| Sim-to-Realギャップが大きい | シミュの物理パラメータが不正確 | Domain Randomizationの範囲を広げる、実機データの割合を増やす |

## 12.9 まとめ

本章では、ロボット学習の主要手法を俯瞰しました。要点は、**ACTは最も手軽で50デモから高性能**、**Diffusion Policyはマルチモーダルに強いが推論が遅い**、**HIL-SERLは人間介入で最高性能を狙えるが手間がかかる**、そして**Cosmos統合でロバスト性を高められる**ことです。実践ではタスクの特性に応じて手法を選び、「ACTで土台→HIL-SERLで改善→合成データで補強」と組み合わせるのが効果的です。

次章では、これらの手法で学習したポリシーを実機にデプロイし、成功率を測って改善するループを回す方法を見ていきます。

# 第13章　推論デプロイと評価・改善ループ
---

これまでで、データを集め（第8章）、ACTなどで学習する（第11章）ところまで来ました。しかし模倣学習は「学習して終わり」ではありません。**学習済みモデルを実機で動かし、成功率を測り、失敗を分析し、足りないデータを足して再学習する**——この改善ループを回すことが、実用的な性能に到達する唯一の道です。本章では、その回し方を整理します。

## 13.1 改善ループの全体像

模倣学習の実力は、次のループを何周回すかで決まります。

```
学習済みモデルをデプロイ → 実機で評価（成功率測定）→ 失敗を分析
→ 不足データを追加収集 → 再学習 →（ループの先頭へ）
```

1周目で成功率が低くても落胆する必要はありません。失敗のパターンを見て「どの状況のデータが足りないか」を突き止め、そこを重点的に足していくのが定石です。

:::message
📷 **画像プレースホルダー**：改善ループの循環図。「デプロイ→評価→失敗分析→データ追加→再学習」を円環で示し、各ステップに対応する章（第11章／本章／第8章）を添える。本章の位置づけを示す図。
:::

## 13.2 学習済みモデルのデプロイ

学習済みモデル（チェックポイント）を読み込ませ、ロボットに自律実行させます。データ収集と同じ `record` コマンドに、`--policy.path` でモデルを指定するのが基本形です。評価結果も映像として保存されるので、後で失敗を見返せます。

```bash
python -m lerobot.record \
  --robot.type=so101_follower --robot.port=<フォロワーのポート> \
  --dataset.repo_id=${HF_USER}/eval_pick_cube \
  --dataset.single_task="Grasp a lego block and put it in the bin." \
  --dataset.num_episodes=10 \
  --policy.path=outputs/train/pick_cube/checkpoints/last/pretrained_model \
  --dataset.push_to_hub=true
```

Colabで学習した場合は、モデルをHugging Face Hubにアップロードしておき、手元のMacにダウンロードして `--policy.path` に指定します（第5章の連携手順）。この評価時、リーダーアームは不要です（ロボットが自分で動くため）。

## 13.3 成功率の測定

評価では、まず十分な回数（10〜20エピソード程度）を自律実行します。成功/失敗の判定は、初期段階では**人が映像を見て手動で記録する**のが確実です。各エピソードの映像を確認し、成功なら1・失敗なら0を記録して集計します。

```python
# 保存された評価エピソードを1つずつ確認して成功率を集計する例
from pathlib import Path
import json

eval_dir = Path(f"~/.cache/huggingface/lerobot/{HF_USER}/eval_pick_cube").expanduser()
episodes = sorted(eval_dir.glob("episode_*"))

results = []
for ep in episodes:
    print(f"{ep.name} の映像: {ep/'video.mp4'}")
    ok = input("成功? (y/n): ").lower() == "y"
    results.append(ok)

rate = sum(results) / len(results)
print(f"成功率: {rate:.1%}（{sum(results)}/{len(results)}）")
json.dump({"success_rate": rate, "details": results}, open("eval.json", "w"), indent=2)
```

より進んだ方法として、HIL-SERL（第12章）で使う**報酬分類器**を流用すれば、成功/失敗を自動判定できます。ただし分類器自体の精度に依存するため、重要な評価では人手確認と併用するのが安全です。

**成功率の目安**は、まず50〜60%を最初のマイルストーンに、実用レベルとしては80〜90%を目指します。1回の評価は最低10エピソード、信頼できる数字が欲しければ20〜30エピソード取ると、偶然のばらつきに惑わされにくくなります。

## 13.4 失敗の分析

改善ループの心臓部が失敗分析です。保存された失敗エピソードの映像を見て、**どのフェーズで、どんな状況で失敗しているか**を分類します。よくある失敗パターンと典型的な原因は次の通りです。

| 失敗パターン | よくある原因 | 対策 |
|---|---|---|
| 物体に近づけない・掴めない | その位置・向きのデータ不足 | 失敗した位置周辺のデモを追加収集 |
| 掴んだ後に落とす | グリッパー制御のデータが少ない、把持が浅い | 掴み〜持ち上げのデモを丁寧に追加 |
| 動作が震える・カクつく | Temporal Ensemble不足、学習不足 | ensembling調整、学習ステップ増 |
| 途中で停止して進まない | 非マルコフ性の学習不足、chunk_sizeが小さい | chunk_sizeを10〜15に、履歴を活用 |
| 特定の色・形だけ失敗 | その見た目のデータ不足 | 該当バリエーションを重点収集 |

ポイントは、「モデルが悪い」と決めつける前に、**まず失敗した状況のデータが十分にあったかを疑う**ことです。多くの場合、失敗は「見たことのない状況」で起きています。

:::message
📷 **画像プレースホルダー**：典型的な失敗パターンを並べた図（掴み損ね／落下／停止／震え）。各パターンの評価映像の一場面と「どのデータを足すべきか」の対応を示し、失敗分析の見方を伝える。
:::

## 13.5 データ追加と再学習

失敗分析で「足りない状況」が分かったら、そこを狙ってデモを追加します。全体をむやみに増やすより、**失敗した位置・向き・対象物のバリエーションを重点的に足す**方が効率的です。たとえば「右奥のブロックだけ掴めない」なら、右奥付近のデモを10〜20エピソード足します。

追加したデータで再学習し（第11章の手順）、再びデプロイして評価します。このループを回すごとに、成功率が階段状に上がっていくのが理想です。1周ごとに「今回はどの失敗を潰すか」を1〜2種類に絞ると、原因と効果の対応が見えて改善が早まります。

## 13.6 デプロイ時の実践的な注意

**環境の一致** — 評価時の照明・カメラ位置・背景・対象物は、データ収集時とできるだけ揃えます。模倣学習は見た目の分布に敏感で、照明が変わっただけで性能が落ちることがあります。

**安全確保** — 自律実行中は予期しない動きをすることがあります。緊急停止（電源オフでモーター脱力）にすぐ手が届く状態で見守り、周囲に壊れやすい物や人の手を置かないようにします（詳細な安全運用は付録も参照）。

**推論の負荷** — SO-101 + ACTはMacのCPU/MPSでもリアルタイム推論が可能ですが、重いモデルや複数カメラで処理が追いつかない場合は、解像度やカメラ台数、chunk_sizeを見直します。

## 13.7 まとめ

本章では、学習済みモデルを実機にデプロイし、評価・改善ループを回す方法を見ました。要点は、**模倣学習は「デプロイ→評価→失敗分析→データ追加→再学習」の反復で伸びる**こと、**成功/失敗はまず人手で確実に記録する**こと、**失敗は多くの場合「データ不足の状況」で起きるため、足りない状況を狙ってデータを足す**こと、そして**評価環境は収集環境と揃える**ことです。この地道なループこそが、デモ映像のような滑らかな動作に到達する近道です。

ここまでで、SO-101で模倣学習を1周し、改善する力が身につきました。次の第3部では、シミュレーションを使ってデータを補い、Sim-to-Realでさらに性能とロバスト性を高める世界へと進みます。
