# はじめに
---

## 「潤沢なGPU環境がなくても、Physical AIは始められる」

ロボットにタスクを学ばせる——強化学習でキューブを掴ませ、模倣学習で人の動きを覚えさせ、シミュレーションで鍛えて実機で動かす。こうしたPhysical AI（物理世界と相互作用するAI）の研究は、長いあいだ「潤沢なGPU環境を持つ研究室のもの」でした。NVIDIA Isaac Simのような本格的なシミュレーターは、動かすだけでハイエンドGPUと大きなVRAMを要求し、環境構築でつまずく人も少なくありません。

本書が題材にする次世代物理シミュレーター **Genesis** は、その常識を変えます。**M4 MacBookで動き、ヘッドレスで動かせるのでGoogle Colabでも簡単に回せる。** 重い並列学習が必要なときだけColabのGPUを借りればよく、RTX 6000やA100専用サーバーのような特殊で高価な環境は要りません。つまり、**M4 Macが1台あれば、誰でもPhysical AIの実験を始められる**——これが本書の出発点です。

## この本の焦点：M4 Mac ＋ Google Colab

本書は「M4 Mac＋必要なときだけGoogle Colab」という、多くの人が手の届く環境を前提に書かれています。Genesisの魅力は、次の2点に集約されます。

- **軽い**：Apple Silicon（M4 Mac）のローカル環境で、物理シミュレーションの動作確認、ロボットアームの操作、可視化、実機制御まで完結できる。
- **ヘッドレスで回せる**：画面表示なしで動くため、Google ColabのようなクラウドGPU上でも簡単に動かせる。重い並列強化学習は、ここに任せればよい。

## 正直な線引き — Macでできること・Colabに任せること

誠実にお伝えします。Genesisの代名詞である「**RTX 4090で1秒間に4,300万フレーム（43 million FPS）**」という驚異的な速度は、NVIDIA GPUでの数字です。**M4 Macでその速度は出ません。** それでも、Macで動くこと自体が革命的です。本書では、次のように役割を分けます。

| やること | M4 Mac | Google Colab |
|---|---|---|
| インストール・物理エンジンの動作確認 | ◯ | — |
| SO-101アームを動かす・IK・可視化 | ◯ | — |
| 実機SO-101の制御（USB接続） | ◯（Macの得意分野） | — |
| 小〜中規模のシミュレーション | ◯ | 大規模はColab |
| **1024並列の強化学習（本書の目玉）** | △（並列数を絞れば流れは体験可） | **◯ 推奨** |
| 大規模な流体・微分可能シミュ・生成パイプライン | △ | ◯ |

Macは「速度を体感する場」ではなく「開発・理解・実機制御の場」、Colabは「重い学習を回す場」。この分担なら、**特殊なGPU環境を持たない人でも、本書のすべてを試せます。**

## 対象読者と前提

- Pythonの基本と、ターミナルでのコマンド操作に抵抗がない方
- ロボット学習・強化学習・Physical AIを、手を動かして学びたい方
- 高価なGPU環境を持たず、M4 Mac（＋無料〜低額のColab）で試したい方

機械学習やロボティクスの専門知識は前提としません。必要な概念はその都度説明します。教材には、部品代約5万円で入手できるオープンソースのロボットアーム **SO-101** を使います。

## 本書の歩き方（難易度マップ）

本書は「動かして理解する」ことを重視し、基礎から応用へ段階的に進みます。

- **基礎編（第1〜5章）**：Genesisとは何かを押さえ、Mac/Colabで環境を作り、物理エンジンの6つのソルバー、ロボット統合の基礎を学び、SO-101をGenesis上で動かします。**すべてM4 Macで完結できます。**

それでは、「もし現実が1万倍速で動いたら」何ができるのか——Genesisの世界を見ていきましょう。

# 第1章　Genesisとは何か — 物理エンジンの再発明
---

本章では、Genesisというシミュレーションプラットフォームが何者で、なぜ今これほど注目を集めているのかを解説する。技術的な詳細に入る前に、まずGenesisの「思想」を理解しておこう。それが分かると、本書全体の設計が一気に見通せるようになる。

## 1.1　誕生の背景 — なぜ既存シミュレーターでは不十分だったか

ロボティクス・AIの研究で、物理シミュレーターはなくてはならない存在だ。実機でロボットを動かす前に、シミュレーター上で何千回・何万回と試行錯誤できる。これが「Sim-to-Real（シミュレーションから実機転用）」の基本的な考え方である。

しかし、既存のシミュレーターにはそれぞれ根本的な制約があった。

| シミュレーター | 強み | 主な制約 |
|---|---|---|
| MuJoCo | 関節・剛体の精度が高い | 速度限界、材料多様性が低い |
| Isaac Sim | GPU並列、フォトリアル | **NVIDIAに依存、要求スペックが高く導入コストも高い** |
| PyBullet | オープンソース、手軽 | 低速、レンダリング品質が低い |
| Gazebo | ROS連携が強い | リアルタイム性能が低い |
| Webots | 教育向けに充実 | 大規模並列学習に不向き |

これらの課題を整理すると、大きく3つの「壁」が見えてくる。**速度の壁**（強化学習には何百万ステップも必要だが、CPUベースは遅すぎる）、**材料の壁**（現実のロボットはやわらかい素材・液体・砂・布とも相互作用するが、既存エンジンの多くは剛体特化で、流体やソフトボディは別ツールになる）、そして**生成の壁**（3Dモデル準備・物理パラメータ調整・報酬設計など、シミュレーション構築に膨大な手作業が必要だった）だ。

そしてもう一つ、本書がとりわけ重視する壁がある。**導入のハードルの壁**だ。たとえばIsaac Simは強力だが、動かすだけでハイエンドのNVIDIA GPUと大きなVRAMを要求し、環境構築でつまずく人も多い。「潤沢なGPU環境を持つ研究室でなければ始められない」——これが、多くの人をPhysical AIから遠ざけてきた現実だった。

**Genesisは、これら4つの壁を同時に突き破るために「ゼロから再設計」されたエンジンである。** とりわけ最後の壁——導入のハードル——を、Genesisは劇的に下げた。**M4 Macのローカル環境で動き、ヘッドレスでも動かせるのでColabでも回せる。** 高価な専用GPUがなくても始められる、という点が、本書がGenesisを選んだ最大の理由だ。

![画像](https://static.zenn.studio/user-upload/4e639f71c252-20260305.gif)

## 1.2　Genesisの4つの顔

Genesisは一言で説明しにくい。一つのツールが複数の役割を担っているからだ。公式ドキュメントは次の4つとして定義している。

**① ユニバーサル物理エンジン** — 剛体・ソフトボディ・流体・弾性体・筋肉など、多様な物質・物理現象を単一フレームワークで統合シミュレートできる。従来は「剛体エンジン」「流体エンジン」「ソフトボディエンジン」がバラバラで、相互作用の再現が困難だった。Genesisはこれらをすべて同一シーン内で扱える。

**② 軽量・高速・Pythonicなロボティクスシミュレーター** — コードは驚くほどシンプルで、Pythonで数行書くだけでロボットアームをシミュレートできる。そして速い。単純なシーンでは**1秒間に4,300万フレーム（43 million FPS）**という前代未聞の速度を実現している（後述）。

**③ フォトリアルレンダリングシステム** — 視覚的に高品質なレンダリングが可能で、深度センサー・RGBカメラ・セグメンテーションなど実機センサーに近い観測データを生成できる。視覚ベースの強化学習やSim-to-Realの精度が上がる。

**④ 生成データエンジン** — 自然言語の記述から、物理的に正確な動画・3Dシーン・ロボットポリシー・アニメーションを自動生成する。Genesisの最も先進的な側面で、VLM（Vision Language Model）を活用したエージェントがシミュレーション構築の手作業を大幅に削減する（第13章で扱う）。

これら4つは独立して使えるが、組み合わせることで真価を発揮する。物理エンジン×生成エンジンで「自然言語指示→並列強化学習→実機転用」というパイプラインが一気通貫で構築できる。

## 1.3　43 million FPS の衝撃 — ただしMacでの現実も正直に

「43 million FPS」がどれほどの意味を持つか、具体的に計算してみよう。

| 指標 | 数値 | 補足 |
|---|---|---|
| リアルタイム速度 | 1×（60 FPS） | 人間が見て自然な速度 |
| Genesis シミュレーション速度 | 43,000,000 FPS | Frankaアーム＋平面の場合 |
| リアルタイム比 | 約430,000倍 | |
| 1時間の学習 = 実時間換算 | 約49年分 | 430,000 × 1h ≈ 49年 |

強化学習に置き換えるとより実感できる。ロボットアームの把持タスクは一般に数百万ステップの学習が必要だが、従来のCPUシミュレーターでは数日〜数週間かかるところ、GenesisのGPU並列なら数十分〜数時間で完了する。「試す→失敗→修正→再試行」のサイクルが劇的に速くなる。

ただし、ここは本書として正直に線を引いておく。**この43 million FPSは、RTX 4090のようなNVIDIA GPU＋シンプルなシーン（Frankaアーム＋平面）での値だ。M4 Macでこの速度は出ない。** シーンの複雑さ（接触数・ソフトボディ・流体）によっても速度は変わる。

では、Macでは意味がないのか——決してそうではない。**むしろ「これだけの性能を持つエンジンが、M4 Macのローカルでも動く」こと自体が革命的だ。** Macでは物理の動作確認・ロボット操作・可視化・実機制御を軽快に行い、43 million FPSの並列性能が本当に欲しい重い学習だけを、ヘッドレスでColabのGPUに任せればいい。本書はこの役割分担を一貫して採る。

## 1.4　統合物理ソルバーのアーキテクチャ

Genesisが多様な物質を扱えるのは、複数の物理ソルバーを統一フレームワーク上に実装しているからだ。各ソルバーの役割を押さえておくと、後の章での選択が容易になる。

| ソルバー | 得意領域 | 具体的な用途 |
|---|---|---|
| 剛体ソルバー | 金属・木材など変形しない物体 | ロボットリンク、建物、床面 |
| MPM（Material Point Method） | ソフトボディ・流体・雪・砂 | ソフトグリッパー、粉体、変形素材 |
| SPH（Smoothed Particle Hydrodynamics） | 液体・流体力学 | 水、液体操作タスク |
| FEM（Finite Element Method） | 弾性体・変形シミュレーション | ゴム、軟組織、弾性構造物 |
| 筋肉モデル（MPM Muscle） | 生物的な筋肉・アクチュエーター | ソフトロボット、虫型ロボット |

重要なのは、これらを「混在」させられることだ。剛体のロボットアーム（剛体ソルバー）が、柔らかいシリコン製グリッパー（MPM）で、ゼリー状の物体（FEM）をつかむ——といったシーンを、一つのシミュレーション内で実現できる（第3章・第10章で詳述）。

## 1.5　CMU・MIT・Stanford・NVIDIAが共同開発した意味

Genesisは2024年12月に発表された。開発陣にはカーネギーメロン大学（CMU）・MIT・Stanford・Columbia・香港大学・清華大学、そしてNVIDIAというそうそうたる顔ぶれが揃う。これはGenesisが単なる研究プロジェクトではなく、**Physical AIの基盤インフラ**として設計されていることを示す。

NVIDIAの参加はGPU最適化とCosmosワールドモデルとの連携（第14章）を、CMU開発の「RoboGen」（自然言語でロボットタスクを自動生成、第13章）の統合は「ロボット学習の自動化インフラ」への志向を、そしてTaichi（Yuanming Huらが開発したGPU数値計算コンパイラ）の技術は43 million FPSを支える根幹を、それぞれ物語っている。

なお、Genesisのロードマップには「生成フレームワークの段階的公開」が明記されている。現時点でオープンソース化されているのは物理エンジンとシミュレーションプラットフォームで、動画生成・3Dシーン生成・ポリシー自動生成などの高度な機能は順次公開予定だ（この現状は第13章で正確に扱う）。

## 1.7　コラム：人間の空間認知とシミュレーター設計

第9章で詳述する「都市シミュレーター」の設計には、認知科学の知見が活きる。人間が空間を理解し移動するとき、次の三段階が働くことが知られている。

| 認知層 | 内容 | システム対応 |
|---|---|---|
| 手続き的知識 | 「駅に行くにはバス通りを北上する」という習慣的ルート | MESA（目的・意図・社会ルール） |
| 地図的知識 | 「街全体の俯瞰的構造」「A地点からB地点の距離感」 | MESA（グラフ探索・経路計画） |
| 視覚ランドマーク知識 | 「コンビニを右折」「花屋の前でふと足が止まる」 | Genesis（物理・一人称観測） |

この三層構造は、MESAとGenesisの役割分担そのものだ。MESAは「意味の世界」（目的・関係性・習慣・社会ルール）を担い、更新頻度は低く、LLMで事前生成した「ペルソナ」として表現される。Genesisは「物理の世界」（衝突・視野・距離・一人称リアリティ）を担い、毎ステップ高速実行される。

重要なのは、**個性やリアリティが現れるのは視覚ランドマーク層（Genesis層）だ**という点だ。同じ「駅に行く」目的でも、Genesis層では各エージェントが異なる行動を見せる——花屋の前で立ち止まったり、工事の音を避けて遠回りしたり。これはルールベースでは表現しきれない「自然なふるまい」であり、強化学習でこそ獲得できる行動だ。**MESAはデータソース、Genesisは実行エンジン。** この分担を守ることで、MESAの豊かなペルソナ資産を活かしつつ、Genesisの並列性能を最大活用できる。

# 第2章　環境構築 — M4 Mac / Google Colab から始める
---

本章のゴールは、Genesisを自分の環境で動かせる状態にすることだ。本書の方針通り、**M4 Mac（Apple Silicon）とGoogle Colabの2つ**を中心に説明する。最後には「Hello Genesis」として、ロボットアームが重力に引かれて動く最初のシミュレーションを体験する。

## 2.1　動作要件 — Macでもここまで動く

Genesisはクロスプラットフォーム対応で、Linux・macOS・Windowsで動く。最大パフォーマンスを求めるならLinux＋NVIDIA GPUが最善だが、**本書はM4 Mac＋Colabを前提**にする。要件の目安は次の通りだ。

| 項目 | M4 Mac（本書の主環境） | Google Colab | 補足 |
|---|---|---|---|
| OS / 環境 | macOS（Apple Silicon） | Colabランタイム | — |
| Python | 3.10〜3.11 | 3.10〜3.11 | venv / conda 推奨 |
| バックエンド | Metal または CPU | CUDA（GPUランタイム） | 後述 |
| 用途 | 開発・可視化・実機制御・小規模実行 | 重い並列学習 | 役割分担 |
| 速度 | 学習の「流れ」を体験できる | GPUで本格学習 | — |

## 2.3　プラットフォーム別の注意点

本書の主環境はM4 Macだが、Colab・Linux・Windowsの勘所も押さえておく。

**macOS（Apple Silicon）— 本書の主環境**：Metal（Appleのグラフィックスフレームワーク）バックエンドで動作する。初期化時に `gs.init(backend=gs.metal)` を指定する。OpenGL/GLFWの競合でビューアーがエラーになることがあるが、その場合は後述のCPUバックエンドやフレーム保存方式で回避できる。並列シミュレーションの速度はNVIDIA GPUより低いが、入門・開発には十分だ。

**Google Colab**：GPUランタイム（T4/A100）でCUDAバックエンドが使える。ただしリアルタイムビューアーは開けないため、フレームを画像保存して動画化する（2.5節）。重い学習はここに任せる。

**Linux / Windows**：Linux＋NVIDIA GPUが最速・最安定。Windowsは動くがやや不安定で、WSL2（Windows Subsystem for Linux）経由でLinuxとして動かすのが安定した選択肢だ。

### よくあるエラーと対処法

| エラー内容 | 主な原因 | 対処 |
|---|---|---|
| ビューアーが開かない（Mac） | OpenGL/GLFWの競合 | CPUバックエンドを試す／フレーム保存方式に切替 |
| `GLIBCXX_3.4.30 not found` | condaのlibstdc++が古い | システムのlibstdc++にシンボリックリンク |
| `trimesh / pymeshlab` エラー | 依存ライブラリ不足 | `pip install trimesh pymeshlab` を別途実行 |
| `CUDA available: False`（Colab） | GPUランタイム未選択 | ランタイムのタイプをGPUに変更 |

## 2.5　Hello Genesis — 最初のシミュレーション

環境が整ったら、最初のシミュレーションを動かそう。Frankaロボットアームを重力下で自由落下させるシンプルなシーンで、「Genesisが動いている」ことを確認する。**Macならビューアーで動きを見られる**。

```python
import genesis as gs

# 1. 初期化（Macなら backend=gs.metal、Colabなら gs.cuda）
gs.init(backend=gs.metal)

# 2. シーンを作成
scene = gs.Scene(
    show_viewer=True,   # Macは表示可。Colabでは False にする
    viewer_options=gs.options.ViewerOptions(
        camera_pos=(2.5, -2.5, 2.0),
        camera_lookat=(0.0, 0.0, 0.5),
        camera_fov=40,
    ),
    sim_options=gs.options.SimOptions(
        dt=0.01,                  # タイムステップ（秒）
        gravity=(0, 0, -9.81),
    ),
)

# 3. エンティティを追加
plane = scene.add_entity(gs.morphs.Plane())                        # 地面
franka = scene.add_entity(
    gs.morphs.MJCF(file='xml/franka_emika_panda/panda.xml'),       # ロボットアーム
    pos=(0, 0, 0),
)

# 4. シーンをビルド（物理計算グラフの構築）
scene.build()

# 5. シミュレーションループ
for i in range(1000):
    scene.step()   # 物理ステップを1回進める（dt=0.01 → 1000回で10秒分）

print("Simulation complete!")
```

コードの要点は5ステップだ。**`gs.init()`** でバックエンド（`gs.metal`＝Apple Silicon、`gs.cuda`＝NVIDIA GPU、`gs.cpu`＝CPU）を指定して初期化する。**`gs.Scene()`** で重力・タイムステップ・ビューアーなどシーン全体を定義する。**`scene.add_entity()`** でオブジェクトを追加する——Genesisでは地面もロボットも流体も「すべてエンティティ」という統一概念で扱う。**`scene.build()`** で物理計算グラフを構築する（build後はエンティティを追加できない）。そして **`scene.step()`** で物理を1タイムステップ進める。

Macでビューアーウィンドウが開き、Frankaアームが重力でゆっくり崩れ落ちていけば成功だ。ウィンドウが開かない・エラーが出る場合は2.3節を参照する。

## 2.6　バックエンドと精度の設定

初期化オプションを押さえておくと、目的に応じた設定ができる。

```python
gs.init(backend=gs.metal)   # Apple Silicon（本書のMac環境）
gs.init(backend=gs.cuda)    # NVIDIA GPU（Colab等）
gs.init(backend=gs.cpu)     # CPU（遅いが環境依存なし・デバッグ向き）
gs.init(backend=gs.gpu)     # 自動選択（cuda か metal を判定）

gs.init(backend=gs.cuda, precision='32')   # f32（デフォルト・高速）
gs.init(backend=gs.cuda, precision='64')   # f64（高精度・やや低速）
```

| 設定 | 速度 | 用途 |
|---|---|---|
| `backend=gs.metal` | 中速 | **M4 Macでの開発・可視化** |
| `backend=gs.cuda` + `precision='32'` | 最速 | Colab等での強化学習・大規模並列 |
| `backend=gs.cuda` + `precision='64'` | やや低速 | 精密な物理解析 |
| `backend=gs.cpu` | 低速 | GPUなし・デバッグ |
