# 3-1. システム全体像
---

改めて確認しておこう。**MESA（Multi-Entity Simulation Architecture）** とは、AIを搭載した住人たちが、それぞれの性格に従って日々の暮らしを送り、互いに関わり合い、意思決定をしながら生活する——そんな3Dの仮想都市を作り出す、自律エージェント・シミュレーションの構想だ。名前の通り「複数の主体（Multi-Entity）」が同じ世界に共存し、一人ひとりが自律的に振る舞うことで、**都市という単位で人間社会の営みそのものを再現する**ことを目指している。単一の賢いAIを作るのではなく、多様な個性を持つエージェントの相互作用から、社会の複雑さや創発を立ち上げようという発想である。

これまでの章では、その MESA が目指す「理想のアーキテクチャ」を描いてきた。5層構造、階層的タスク分解、DINOv2によるセグメンテーション、V-JEPA2による物理予測——どれも壮大な設計図だ。

ここからの第3部では視点を変える。**実際に手を動かして作った実装**を見ていく。題材は、**筆者（oggata）自身が開発・公開しているリポジトリ `mesa-clouds-renderer`** だ。理想像のすべてを実装しているわけではない。むしろ驚くほど絞り込まれている。しかしその絞り込み方にこそ、「動くフィジカルAI」を作るときに何が本質で何が後回しにできるかのヒントがある。

## なぜ、これを作ろうと思ったのか

第2部までで描いてきた「理想のアーキテクチャ」は、正直に言えば、読んでいて壮大すぎて手が止まる。VLMもV-JEPA2も継続学習も、どれも魅力的だが、「で、結局まず何から作ればいいのか？」という問いには答えてくれない。筆者自身、第2部のような設計図を眺めながら、ずっとその問いを抱えていた。

もう一つ、個人的にずっと作りたかった世界観がある。人間の脳には役割分担がある。たとえば買い物に出かけるとき、まず頭の中で俯瞰した地図を思い浮かべて「あの店に行こう」と大まかな計画を立て、いざ歩き出したあとは、目に映る景色を頼りに進んでいく。地図を一歩ずつなぞっているわけではない。そして道の途中で「あ、こっちに新しい店ができてる」と目に入れば、その場で行き先の計画そのものを書き換えたりもする。**計画を立てる脳と、目で見て実行する脳、そして見たものが計画を上書きしていく——この一連の流れを、NPCで再現したい**というのが、ずっと抱えていた思いだった。

言い換えれば、目で見たものを捉え、記憶し、頭の中の地図と実際の地図をすり合わせながら動く。移動は単に経路をなぞるだけでなく、視界に入ったものによって、その先の計画自体が変わりうる——そんな世界だ。

そこで筆者は、理想の全部を一度に作ろうとするのをやめた。代わりに、この世界観の**核となる「目で見た景色から次の一歩を決める」部分**——脳でいえば計画ではなく実行にあたるところ——を一点に絞り、実際に動く最小のシステムとして作ることにした。それが `mesa-clouds-renderer` だ。ブラウザで誰でも動作を見られる形（mesa-clouds-renderer.onrender.com）で公開しているのも、「理想を語る」だけでなく「動くものを見せる」ことにこだわりたかったからだ。

だからこの第3部は、完成された模範解答の解説ではない。**理想と現実のあいだで、筆者が実際にどこを削り、どこを残したか**の記録だ。その取捨選択の判断こそが、これから自分でフィジカルAIを作ろうとする読者にとって、いちばん役に立つ部分だと信じている。

## このプロジェクトが解いている問題

一言で言うと、**「人間が買い物に出かけるときの脳の役割分担を、NPCで再現する」** というプロジェクトだ。前節で述べた通り、人は俯瞰した地図で大まかな計画を立て、実際の移動は目に見えた景色を頼りに行い、そして見えたものによって計画を書き換えていく。この「計画」と「視覚による実行」、そして「視覚が計画を上書きする」流れを、シミュレーション内のエージェントで再現したい。

多くのゲームAIやシミュレーションでは、エージェントは自分の座標 `(x, y)` を知っていて、目的地までの経路をA*などで一気に計算し、あとはその線をなぞるだけだ。効率的ではあるが、これは脳でいえば「計画」だけで移動を済ませてしまっている状態で、「目で見て動く」という実行の脳が存在しない。だから、途中で景色に何かを見つけて計画を変える、といった柔軟さも生まれない。

そこでこのプロジェクトでは、**移動の判断を座標ではなく「目に見えた景色」に委ねる**。エージェントは一人称視点の映像だけを見て、次の一歩（前進/左/右）を決める。これは、脳の役割分担のうち「視覚による実行」にあたる部分を、あえて切り出して作ったものだ。そして後の節（3-6）で見るように、外から「行き先」を指示できる仕組み（目標条件付け `z`）も用意してあり、これが将来的に「計画を立てる脳（LLMなど）」と「実行する脳（このポリシー）」をつなぐ足場になる。

その「視覚で動く」中身を、1ステップの流れで表すとこうなる。

```
エージェントの (x, y, 向き)
  └─ 一人称視点をレイキャスタで描画（テクスチャ付き 224×224）
       └─ DINOv2 → CLS token (384次元)  =「今どんな景色か」の要約ベクトル
            └─ ペルソナ別ポリシー(persona_X.onnx) → 行動(前進/左/右)
```

エージェントは自分の座標をポリシーに渡さない。渡すのは「今見えている画像をDINOv2に通した384次元のベクトル」だけだ。それでも、探索好きなペルソナは広く歩き回り、グルメなペルソナは飲食店に寄っていく。座標を持たないのに、目的地に「見た目で」たどり着く。

:::message
📷 **画像プレースホルダー**：配信デモ（mesa-clouds-renderer.onrender.com）のスクリーンショット。都市全景を俯瞰し、複数のペルソナエージェントが同時に歩き回っている様子が伝わるもの。
:::

## 学習と本番で「同じ観測」を使う

この手のプロジェクトで最もハマりやすい落とし穴は、**学習時とデプロイ時で観測の作り方が微妙に違う**ことだ。学習はPythonのシミュレータで、本番はブラウザやサーバーのJavaScriptで動く、というように実装言語が変わると、レンダリングのアルゴリズムに些細な差異が生まれやすい。角度の計算が1行違うだけでも、学習済みモデルが「見たことのない画像」を本番で受け取ることになり、行動が崩壊する。

このプロジェクトはここに一貫性を持たせることを最優先の設計方針にしている。

| | 学習（Colab / PyTorch） | 本番（サーバー / Node.js） |
|---|---|---|
| FPV描画 | テクスチャDDAレイキャスタ（GPU一括） | テクスチャDDAレイキャスタ（JS） |
| 観測 | DINOv2 → CLS(384) のみ | DINOv2 → CLS(384) のみ |
| 種類認識 | テクスチャをDINOv2が見て学習 | テクスチャをDINOv2が見て推論 |
| 移動可否 | マップ配列（道路/建物が通行可） | マップ配列（`PASSABLE`） |
| 好み（例:グルメ） | `food_bonus` 報酬で学習 | `persona_X.onnx` の重みに宿る |

**DDA（Digital Differential Analyzer）レイキャスタ**というアルゴリズムを、学習用のPyTorchコードと本番用のJavaScriptコードの両方に、ほぼ同じロジックで実装し直している。これは車輪の再発明のようだが、「学習と本番で観測がズレない」という制約を満たすには、実は最も確実な方法だ。共通ライブラリを介する方法もあるが、GPU一括処理（学習）とリアルタイム逐次処理（本番）という要求が違う以上、アルゴリズムだけを移植する方が現実的だった、と読み取れる。

## パイプライン全体の流れ

```
[学習: Colab] build_pro_onnx_by_persona.ipynb
   テクスチャFPV → DINOv2 → CLS → PPO学習 → persona_A〜E.onnx
                                          → dinov2_vits14.onnx (本体エクスポート)
        │  data/ にコピー
        ▼
[本番: server.js]  headless-gl + Three.js
   毎ステップ: 各エージェントのFPV → dinov2_vits14.onnx → CLS → persona_X.onnx → 行動
   映像: Three.jsで都市を描画 → JPEG → WebSocket → ブラウザ
```

学習側は1つのノートブック（`build_pro_onnx_by_persona.ipynb`）に完結している。GPU上でPPO（Proximal Policy Optimization）を回し、ペルソナごとの重みをONNX形式でエクスポートする。本番側はNode.jsの単一プロセス（`server.js`、約1,100行）で、`headless-gl`によるCPUレンダリングと`onnxruntime-node`による推論を両方担っている。

ここで重要なのは、**重い「見る」部分（DINOv2）は全ペルソナで凍結・共有し、学習で育てるのは末端の小さなヘッド（`persona_X.onnx`）だけ**という設計だ。DINOv2の本体は約84MBあるが、これを5人分のペルソナごとに複製するのではなく、1つのONNXセッションを共有し、その出力（CLS384次元）だけをペルソナ別の軽量な全結合層に渡す。これにより、ペルソナを何人追加してもメモリコストはほぼ増えない。

:::message
📷 **画像プレースホルダー**：`build_pro_onnx_by_persona.ipynb`（Colab）のセル一覧、または目次のスクリーンショット。学習パイプラインが1本のノートブックに完結していることが伝わるもの。
:::

# 3-2. 「見て動く」観測 — テクスチャDDAレイキャスタでFPVを描く
---

エージェントの「目」にあたる部分——FPV（一人称視点）画像をどう作るか——から見ていく。

## なぜ古典的なレイキャスタなのか

3D都市を描画するだけなら、Three.jsでカメラを1台置いてレンダリングすればいい。実際、ブラウザに配信する俯瞰映像はThree.jsで描かれている。しかし、**エージェントが「見て」判断するための観測画像**は、あえて別の軽量な仕組みで作られている。それが1990年代のFPSゲーム（Wolfenstein 3Dなど）で使われた**DDAレイキャスタ**だ。

理由は単純で、数百〜数千体のエージェントが毎ステップ「自分視点の画像」を必要とするとき、フル3Dレンダリングをその数だけ回すのは計算コストが高すぎる。DDAレイキャスタは、画面の各列（x座標）につき1本のレイをグリッド上でトレースするだけで壁との交差点を求められるため、非常に軽量だ。学習時は数千環境を並列でGPU一括処理し、本番はNode.js上でエージェント数分を逐次処理する——どちらの用途にも、この軽さが効いてくる。

## DDAレイキャスタの仕組み

`server.js` の `renderFPImageCfg` 関数（`server.js:265`）が本番側の実装だ。核となるループはこうなっている。

```js
const dirX=Math.cos(agent.th), dirY=Math.sin(agent.th);
const pl=Math.tan(FOV/2), planeX=-dirY*pl, planeY=dirX*pl;
for(let x=0;x<W;x++){
  const cam=2*x/W-1;
  const rdx=dirX+planeX*cam, rdy=dirY+planeY*cam;
  let mapX=Math.floor(agent.x), mapY=Math.floor(agent.y);
  const ddx=rdx===0?1e30:Math.abs(1/rdx), ddy=rdy===0?1e30:Math.abs(1/rdy);
  // ... stepX/stepY, sdx/sdy を初期化 ...
  while(g++<64){
    if(sdx<sdy){sdx+=ddx;mapX+=stepX;side=0;}else{sdy+=ddy;mapY+=stepY;side=1;}
    if(map[mapX][mapY]===BUILDING){ /* ヒット */ break; }
  }
}
```

画面の横方向（224ピクセル）1列ごとに1本のレイを飛ばし、グリッドのマス目境界を跨ぐたびに「次にどちらの軸の境界に先に到達するか」（`sdx`と`sdy`の比較）を判定しながら進む。これがDDAの本質で、浮動小数点の刻み幅を細かくとる愚直なレイマーチングより高速かつ正確だ。

壁（建物セル）にヒットしたら、そのセルの建物タイプに対応するテクスチャ（後述）から該当する列のピクセルを抜き出し、距離に応じた明るさ補正をかけて画面に描く。ヒットしなければ、上半分は空の色、下半分は地面の色で塗る。

```js
const br=Math.min(1.0, Math.max(0.35, 1.0-perp/9));
```

遠いほど暗くなるシンプルな距離フォグだ。これによりDINOv2に渡す画像にも「奥行き感」が残る。

## テクスチャは学習・本番で共有

`textures/` ディレクトリに配置された建物テクスチャ画像（牛丼屋、ラーメン屋、オフィスなど8種類）は、リポジトリに同梱されており、学習用ノートブックと本番サーバーの両方から**まったく同じファイル**が読み込まれる。

```js
  await Promise.all(BLDG_TYPES.map(async (bt,i)=>{
    const fp=path.join(__dirname, bt.textureFile);

    // ...
  }));
}
```

`sharp` ライブラリで64×64にリサイズしたテクスチャをメモリに保持し、レイがヒットした建物の種類とテクスチャ座標（UV）に応じてピクセルをサンプリングする。学習側（Colab）もPyTorchで同じテクスチャを同じ解像度で読み込む。**このテクスチャの一致こそが、次章で説明するDINOv2による建物認識の土台になる。**

## 座標ではなく画像を「観測」として扱う設計の意味

`renderFPImageCfg` が返すのは、あくまで224×224×3のRGB画像（CHW形式、0〜1に正規化済み）であって、エージェントの座標や建物の意味ラベルではない。この画像がそのままDINOv2に渡され、そこから初めて「これは牛丼屋っぽい景色だ」という意味が抽出される（3-3章）。

つまりレイキャスタの役目は「見えている物理的な形と色を正確に再現すること」までで、「それが何であるか」の認識は一切担っていない。認識と描画をこの段階できっぱり分離しているのが、このプロジェクトの設計上のポイントだ。

# 3-3. DINOv2で建物を「見分ける」
---

前章で作った224×224の画像を、どうやって「意味」に変換するか。ここがこのプロジェクトの核心部分であり、第2部で説明した「Vision Model」（2-6章）の理論を、実際に組み込んだ実例にあたる。

## one-hotラベルではなく埋め込みベクトルを選んだ理由

素朴に「牛丼屋を見分けたい」と考えると、8種類の建物タイプに対する分類器（CNN + softmax）を学習し、one-hotベクトルとして扱いたくなる。実際、このリポジトリにも `building_classifier.onnx` という分類器は存在する（3-5章で触れる）。しかし、ポリシーの主要な観測として使われているのは、**分類結果ではなく、DINOv2が出力するCLSトークン(384次元)そのもの**だ。

README（`README.md:238`）にはこう明記されている。

> 建物を「クラスの枠（one-hot 8個）」で表すと、新カテゴリ追加＝枠追加＝次元変化＝**全モデル作り直し**になり硬い。DINOv2のCLSは、RGBが任意の色を3数字で表すように、任意の見た目を384数字で表す連続ベクトル。新しい建物は「新しい座標」になるだけで次元は不変＝**オープン語彙で柔軟**。

これは実務上とても大きな違いを生む。one-hot分類器方式では、新しい建物タイプ（例えば「本屋」）を追加するたびに、出力次元が変わり、下流のポリシーネットワークをすべて作り直す必要がある。埋め込みベクトル方式なら、DINOv2は凍結されたまま何も変わらず、新しい建物のテクスチャを見せれば、384次元空間の「新しい座標」にただ配置されるだけで済む。

## 実際にどれくらい「見分けられて」いるのか

DINOv2は意味の近いものを近くに配置する埋め込み空間を持つ。README ではこれを実際に検証している。8種類の建物テクスチャをDINOv2に通してCLSの類似度を測ると、**飲食店（牛丼・ラーメン・弁当・カフェ）どうしが自然に固まる**ことが確認できる。

```
- 飲食店4種どうしの平均類似度：57
- 飲食店 × 非食（オフィス等）の平均類似度：41
```

「飲食店」というカテゴリを、人間が明示的にラベル付けして教え込んだわけではない。DINOv2が自己教師あり学習で獲得した汎用的な視覚表現の中に、たまたま「見た目が似ているものは近くに配置される」という性質があり、それが結果的に人間にとって意味のあるクラスタ（飲食店グループ）を作っている。これが2-6章で説明した自己教師あり学習の実践的な効能だ。

ただし完全にクリーンな分離ではない。牛丼屋と病院が見た目の色使いなどで近くなる、といったノイズも実際に確認されている。README はこの点を率直に認めている。

> 完全にクリーンな分離ではありません（牛丼屋と病院が見た目で近い等）。本プロジェクトは精度より「DINOv2的アプローチで世の中のものを見分けて行動できるか」の検証を主眼にしています。

**精度を追い求めるプロジェクトではなく、「ラベルなしの汎用視覚表現だけでどこまで行動を分岐できるか」を確かめる実験** という位置づけが明確にされている点は、実装を読み解く上で重要な補助線になる。

:::message
📷 **画像プレースホルダー**：Colabで生成した8種類の建物テクスチャ間のCLS類似度ヒートマップ。飲食店4種（gyudon/ramen/bento/cafe）どうしのマスが濃く固まって見えるもの。
:::

:::message
📷 **画像プレースホルダー（任意）**：CLSベクトルをt-SNEやPCAなどで2次元に落とした散布図。8クラスの建物テクスチャがどう配置されるかが視覚的にわかるもの。

## 1ステップでの処理の流れ

建物認識が実際にどのタイミングで完了するのか、README（`README.md:74`）に沿って整理するとこうなる。

```
① 状態        各エージェントの (x, y, 向きθ)
② FPV描画     θ方向の一人称視点をDDAレイキャスタで224×224×3に描画
③ DINOv2      dinov2_vits14.onnx (ImageNet学習済み・凍結) に通す。出力は2つ:
              ・cls   (384)      = 画像全体の意味要約 =「今どんな景色/建物か」
              ・patch (256×384)  = 空間トークン (seg_head用)
④ 建物認識    ここで"認識"は完了している。明示的な分類器は必須ではない
              (DINOv2の埋め込み自体が認識)
⑤ 入力組み立て ポリシーへ渡すベクトルを作る
⑥ ポリシー    persona_X.onnx → logits(3) → softmax → サンプリングで行動決定
```

ここで着目したいのは④の「ここで"認識"は完了している」という一文だ。分類器の出力（softmaxの確率）を待つ必要はなく、CLSベクトルそのものがすでに「認識結果」として機能している。分類ヘッドは、あくまでオプションの追加情報として後段に足せる部品にすぎない（3-5章の392次元モデルで実際に使われる）。

# 3-4. PPOによるペルソナ別ポリシー学習
---

DINOv2が「見る」役を担うなら、「見た結果からどう動くか」を決めるのがポリシーネットワークだ。この章では、`persona_X.onnx` がどうやって作られるのかを見ていく。

## 学習の構造：重いDINOv2を凍結し、軽いヘッドだけ学習する

ポリシーネットワーク自体は非常に小さい。DINOv2が出力するCLS(384次元、goal条件付きなら392次元)を受け取り、全結合層（FC）を経てActor（行動確率）とCritic（価値関数）を出力するだけの構造だ。

```
CLS(384) → FC層 → Actor → 行動logits(3: 前進/左/右)
                 → Critic → 状態価値
```

学習アルゴリズムはPPO（Proximal Policy Optimization）——方策勾配法の中でも安定性が高く、実装の手間が比較的小さいことから広く使われている強化学習アルゴリズムだ。学習ノートブック（`build_pro_onnx_by_persona.ipynb`）では、数千の並列環境（`N_ENVS`）を1つのGPU上でバッチ処理し、ロールアウトを集めてはPPOの更新を回す、という典型的な構成になっている。

DINOv2の重みは学習全体を通して**凍結**されている。勾配はDINOv2までは流れず、末端のFC層とActor/Criticヘッドだけが更新される。これにより:

- 学習が軽い（更新するパラメータ数が桁違いに少ない）
- 5人のペルソナを学習しても、DINOv2は1回だけロードすればいい（実際には各ペルソナの学習を独立に回すが、重い視覚表現部分は使い回せる）
- 本番でも同じ理屈でDINOv2セッションを1つだけ共有できる（3-1章で触れた通り）

:::message
📷 **画像プレースホルダー**：Colabの学習曲線（報酬の推移をエピソード/ステップ数に対してプロットしたグラフ）。ペルソナごとの収束の様子が比較できると理想的。
:::

## ペルソナごとの「性格」はどこに宿るか

5人のペルソナ（A〜E）は同じネットワーク構造・同じDINOv2を使うが、**学習時の報酬関数が異なる**ことで、最終的に異なる重みに収束する。README はこれを明確に述べている。

> ハードコードのルールはどこにもない。そのペルソナの `persona_X.onnx`（学習済みの重み）の中にある。

例えば「グルメなペルソナ」を作りたい場合、`persona_rewards.json` にたった2行を追加するだけでいい。

```jsonc
"food_bonus": 8.0,
"food_classes": [0, 1, 2, 3]   // gyudon, ramen, bento, cafe
```

これは `dinov2seg/METHOD_A_food_persona.md` に手順が残されている「方法A」の実装だ。学習環境（`PersonaVecEnv`）の `step()` 関数の中で、エージェントが到着したセルの建物タイプが `food_classes` に含まれていれば、追加のボーナス報酬を与える。

```python
# ── グルメ報酬: 飲食店タイプに到着したら追加ボーナス ──
arr_bt  = BTYPE_T[ri2, ci2]                       # 到着セルの建物種類
is_food = torch.zeros_like(arrived)
for fc in rp.get('food_classes', []):
    is_food = is_food | (arr_bt == int(fc))
rew = torch.where(arrived & is_food, rew + float(rp.get('food_bonus', 0.0)), rew)
```

この報酬は**学習時にだけ**効く。実行時（本番のサーバー推論）には報酬という概念自体が存在しない。学習によって「飲食店に寄ると得だった」という経験が、FC層の重みという形に焼き込まれ、本番ではその重みが前進/左/右の確率分布としてそのまま出力される。README の言葉を借りれば「食い意地が良い」という性格は、コードのどこにもif文として存在せず、数値化された重みの中に溶け込んでいる。

:::message
📷 **画像プレースホルダー**：グルメ設定ON/OFFでのエージェントの軌跡比較。同じマップ上で、food_bonusありのペルソナが飲食店周辺に集まる様子とそうでないペルソナの動線を並べたもの（standaloneのTrail表示などを利用）。
:::

## 建物種類の伝え方：学習は擬似分類器、本番は本物の分類器

`METHOD_A_food_persona.md` を読むと、興味深い非対称性がある。学習時には、シミュレータが「正解」の建物タイプ（`BTYPE_T`）を直接知っているため、one-hotベクトルとして与えている。

```python
def building_in_view(xs, ys, ths):
    """中央レイで最初に当たる建物の種類をone-hot(N,8)で返す。学習時の擬似分類器。"""
```

一方、本番のサーバーには「正解」が存在しない（エージェントは本当にカメラ画像しか持たない設定なので）。そこで本番では `building_classifier.onnx` にDINOv2のCLSを通し、softmaxで求めた確率をone-hotの代わりに使う。

```js
// server.js: inferAction 内
}else if(meta.inputSize>cls.data.length && bldgSession){
  const bo=await bldgSession.run({[bldgIn]:new ort.Tensor('float32', cls.data, [1, cls.data.length])});
  const bl=bo[bldgOut].data;
  // softmax → probs
  cat.set(probs, cls.data.length);
```

学習は「正解one-hot」、本番は「分類器のsoftmax確率」——この2つは厳密には同じ分布ではない。ドキュメントはこのギャップを隠さず明記している。

> 学習=one-hot / 本番=分類器softmax … 種類表現が厳密には違う（one-hot vs 確率）。DINOv2+FCで吸収できる範囲だが、寄り方が不安定なら本番の分類器精度を上げるのが効く。

理論上の理想（学習と本番の完全な一致）と、実際に組める現実的な妥協点（DINOv2の表現力に押し付けて吸収させる）の間で、意図的にバランスを取っていることがわかる。

# 3-5. ONNXデプロイと学習/本番の一貫性
---

学習(PyTorch/Colab/GPU)と本番(Node.js/CPU)は、実行環境もフレームワークもまったく異なる。この橋渡しをしているのが**ONNX**（Open Neural Network Exchange）だ。この章では、モデルがどうエクスポートされ、サーバー側でどうロードされるかを見ていく。

## 何がONNX化されているか

`data/` ディレクトリに配置されるファイル一式がそれにあたる。

```
data/
├── dinov2_vits14.onnx (+.data)     ← DINOv2本体（全ペルソナ共有・1個・約84MB）
├── seg_head.onnx (+.data)          ← セグメンテーションヘッド（任意）
├── building_classifier.onnx (+.data) ← 建物分類ヘッド（任意）
├── persona_A.onnx + persona_A_meta.json
├── ... (B〜E)
```

DINOv2本体はMeta公式のViT-S/14の重みを、学習ノートブック内でONNXエクスポートしたものだ。学習しているわけではなく、あくまで**凍結した「汎用の目」**として使い回されるコンポーネントであることは3-3章で述べた通り。

各ペルソナのONNX(`persona_X.onnx`)は非常に軽量で、対応する`_meta.json`とペアで配置される。このmetaファイルが、サーバー側の挙動を大きく左右する。

## metaファイルによる自動判別の仕組み

サーバー起動時、`loadOnnxSessions` 関数（`server.js:193`）が各ペルソナのmetaを読み込み、観測方式を自動判別する。

```js
personaMeta[p.id]={
  inputSize: isize,
  goalDim: m.goal_dim||0,             // >0 ならgoal条件付け(cls+z)。0=従来(clsのみ)
  dino: isize!==iw*ih*ic,             // 生画像サイズと違う → DINOv2方式
  cfg:{ w:iw, h:ih, fov:(m.fov_deg||60)*Math.PI/180, ... },
};
```

コメントにある通り、`input_size` の値によって観測方式が切り替わる。

| `input_size` | 意味 |
|---|---|
| 12288 (= 64×64×3) | 旧CNN方式。生のFPV画像をそのままヘッドに入力（レガシーのペルソナE用） |
| 384 | DINOv2方式。CLSのみ |
| 392 | DINOv2方式。CLS(384) + 建物分類8種、または CLS(384) + goal z(8) |

同じ392という次元でも、`goal_dim` フィールドの有無によって「建物分類の確率が付いているのか」「目標ベクトルzが付いているのか」を区別する。これはコードのコメントにも明記されている興味深い分岐だ。

```js
//   goal条件付け(meta.goalDim>0): [cls(384), z(goalDim)]
//   それ以外: CLSのみ(384) か CLS+建物分類(392)(legacy)
```

つまり **モデルファイルを差し替えるだけで、サーバー側のコードは一切変更せずに新しい観測方式に対応する**設計になっている。新しいペルソナを追加するときも、metaファイルさえ正しく吐き出せば、`server.js` は自動的に正しい経路を選ぶ。

## 起動時の検証ログ

`ortSessions[p.id]=await ort.InferenceSession.create(op,...)` でセッションを作ったあと、サーバーはダミー入力で1回テスト推論を実行し、成功したモデルのみを有効化する。

```js
await ortSessions[p.id].run({[nm]:new ort.Tensor('float32',new Float32Array(dim),[1,dim])});
console.log(`[ONNX] persona_${p.id} OK  ${mode}`);
```

これにより、起動ログを見るだけで「今どのモデルがどの経路で動いているか」が一目でわかる。

```
[Raycast] textures 8/8 loaded ready=true
[ONNX] dinov2_vits14 OK
[ONNX] persona_A OK  DINOv2(384)   ← 384=goalなし旧モデル
[ONNX] persona_B OK  DINOv2(392)   ← 392=goal対応 or 建物分類付き
```

もしONNXの読み込みに失敗したり、対応するファイルが存在しなければ、エージェントは `biasedRandom` 関数——前方に障害物がなければ高確率で前進する単純なランダム行動——にフォールバックする。「モデルが壊れていてもサーバー全体がクラッシュしない」という頑健性が、この二段構えの設計から読み取れる。

:::message
📷 **画像プレースホルダー**：`node server.js` 起動時のターミナル出力。`[Raycast] textures 8/8 loaded`・`[ONNX] dinov2_vits14 OK`・`[ONNX] persona_A OK DINOv2(...)` のログが並んでいる様子。
:::

## モデルの差し替えだけで観測方式を変えられる

ここまでの設計をまとめると、`server.js` 自体は「どんな観測方式にも対応できる汎用の推論エンジン」であり、**具体的な挙動（384か392か、goal対応かどうか）はすべてONNXファイルとmetaファイルの中に閉じ込められている。** 学習ノートブック側で新しい構成のモデルを吐き出せば、サーバーのコードを一行も触らずに新しい機能が有効になる——次章で扱う「目標条件付け(z)」がまさにこの仕組みを使って実現されている。

# 3-6. 目標条件付け（z）とLLM連携の足場
---

これまでのペルソナは、景色（CLS）だけを見て自律的に動く。ここに**目標ベクトルz**を足すと、「病院へ向かえ」「飲食店へ寄れ」といった行き先を外部から指示できるようになる。これは第1部・第2部で繰り返し語られてきた「LLMが上位の意思決定を担い、下位のポリシーが実行する」という階層構造(2-3章、2-5章)の、実際に動く最小構成にあたる。

## 入力を拡張するだけで実現する

```
ポリシー入力 = [ cls(384), z(8) ] = 392
   z = 行き先の建物タイプ one-hot(8)  (gyudon, ramen, ..., hospital)
   z が全ゼロ = 「目標なし」= 従来どおりの自律行動
```

ポイントは、**ポリシーネットワーク自体は書き換えない**ということだ。z対応モデルを学習する際、CLS(384)にz(8)を連結した392次元を入力として訓練する。z がすべてゼロのときは「目標なし」を意味し、この状態を含めて学習することで、zを渡さなければ従来通りの自律的な挙動になる。

```python
GOAL_NONE_PROB = 0.4   # 既定: 4割の確率でz=ゼロを混ぜて訓練
```

学習時に一定確率でz=ゼロのサンプルを混ぜておくことで、「目標が指定されないケース」に対しても崩壊しないポリシーが育つ。これが後方互換性の鍵になっている。

## サーバー側の受け口はスロットの書き換えだけ

サーバー側の実装（3-5章で見た `inferAction` 関数）を思い出すと、z の扱いはこう書かれている。

```js
if(meta.goalDim>0){
  inDim=cls.data.length+meta.goalDim;
  const cat=new Float32Array(inDim);
  cat.set(cls.data,0);
  const z=agent.goalZ;     // Float32Array(goalDim) をセットすれば誘導できる
  if(z && z.length===meta.goalDim) cat.set(z, cls.data.length);
  inData=cat;
}
```

`agent.goalZ` という1つのフィールドを外部から書き換えるだけで、次の推論ステップから行動が変わる。重みは一切更新しない。**推論時に読み込む入力スロットを外から差し替えているだけ**であり、いわば「毎回同じ関数に、引数を1つ増やして渡している」に近い単純さだ。

この単純さこそが、LLMのような外部プロセスとの連携を容易にしている。LLMは複雑な学習も推論もこのポリシーに対して行う必要がなく、ただ「このエージェントの行き先はここだ」という情報を `agent.goalZ` に書き込むだけで、下位の運動制御には一切関与せずに指示を伝えられる。

## z を設定する3つの経路

README（`README.md:135`）によれば、z の設定には3つの入り口が用意されている。

| 経路 | 使い方 |
|---|---|
| standalone UI | 画面下「🎯 Goal」ボタンでペルソナ別に行き先の建物タイプを選択 |
| standalone JS | `setGoal('A', 7)`（病院へ）/ `setGoal('A', -1)`（解除） |
| server HTTP | `GET /goal?persona=A&type=7`（設定）/ `type=-1`（解除） |

HTTPの `/goal` エンドポイントの実装は `server.js:938` にある。

```js
if(urlPath==='/goal'){
  const pid=(q.get('persona')||'').toUpperCase();
  const type=parseInt(q.get('type'),10);
  const a=agents.find(ag=>ag.def.id===pid);
  const gd=(personaMeta[pid]&&personaMeta[pid].goalDim)||0;
  if(isNaN(type)||type<0){ a.goalZ=null; }
  else { const z=new Float32Array(gd||8); if(type<z.length) z[type]=1; a.goalZ=z; }
  res.end(JSON.stringify({ok:true, ..., note:gd>0?'applied':'保持のみ...'}));
}
```

これは単なるREST APIであり、シェルからcurlで叩くこともできれば、将来的にLLMのツール呼び出し（Tool Use / Function Calling）の出力として直接叩かせることもできる。「LLMをつなぐ場合は、この`agent.goalZ`（または`/goal`）をLLMが書き込む口にする」とREADMEにも明記されている通り、この実装は**LLM統合そのものではなく、LLM統合のための「足場」**として位置づけられている。

:::message
📷 **画像プレースホルダー**：standalone UI（`?admin=1`）の「🎯 Goal」ボタンとペルソナ別の行き先選択パネルのスクリーンショット。
:::

## 384モデルとの後方互換

まだ384次元（goal非対応）のモデルを使っているペルソナに `/goal` でzを送っても、サーバーはエラーにはならない。値は `agent.goalZ` に保持されるが、`meta.goalDim` が0のため推論には反映されない。APIのレスポンスにもその旨が明示される。

```json
{"ok":true, "note":"保持のみ(このpersonaは384モデル。392に差し替えると有効)"}
```

392モデルに差し替えた瞬間、同じUIやAPIがそのまま効くようになる。「まずは自律行動のペルソナとしてリリースし、あとから目標追従に対応させる」という段階的な移行を、既存のインターフェースを壊さずに実現できる設計になっている。

:::message
📷 **画像プレースホルダー**：goal設定前後でのエージェント軌跡の比較。「病院へ」を指示する前は自律的にランダムに歩いていたエージェントが、`/goal?persona=A&type=7` 実行後に病院へ向かって進路を変える様子（2枚並べる、またはTrail表示のビフォーアフター）。
:::

# 3-7. 配信インフラ
---

最後に、ここまで見てきたシミュレーションを、実際にブラウザで見られる形にする配信の仕組みを見ていく。

## サーバーサイドレンダリングという選択

このプロジェクトの最大の特徴は、**3D都市のレンダリングをブラウザではなくサーバー側で行っている**ことだ。`headless-gl` というNode.js用のヘッドレスOpenGL実装を使い、Three.jsのシーンをGPUなしのサーバー環境でも描画できるようにしている。

```js
const gl = require('gl');
const THREE = require('three');
// ...
function createRenderer(){
  // headless-gl の WebGL コンテキストで THREE.WebGLRenderer を初期化
}
```

サーバーが毎ティック、都市全景を描画してJPEGにエンコードし、WebSocketで接続中の全クライアントにブロードキャストする。クライアント（`client.html`）はJPEGフレームを受信して `<img>` や `<canvas>` に流し込むだけの、非常に薄いビューアになっている。

```
Three.jsで都市を描画 → JPEG → WebSocket → ブラウザ
```

この構成の利点は、クライアント側の性能に依存せず、どんな非力な端末でも同じ映像が見られることだ。欠点は、`headless-gl` がCPUレンダリングであるため、解像度やFPSがそのままサーバー負荷に直結することだ。README にもその点への注意書きがある。

> `headless-gl` はCPUレンダリングのため、解像度・FPSがそのまま負荷になります。

## 配信は3系統

`server.js` のHTTPハンドラ（`server.js:924`以降）を見ると、用途の異なる3つのエンドポイントが用意されている。

| URL | 内容 |
|---|---|
| `/`（`client.html`） | WebSocketでJPEGフレームを受信する軽量ビューア |
| `/standalone.html` | ブラウザ単独版。DINOv2/personaをブラウザ内で直接推論 |
| `/goal` | 目標zの設定・一覧API（3-6章） |

興味深いのは `/standalone.html` の存在だ。これはサーバーサイドレンダリングとは対照的に、**DINOv2とONNXモデルをブラウザの中で直接動かす**実装になっている。`data/`と`textures/`はサーバーから静的配信されるが、推論そのものはクライアントのブラウザで完結する。同じシミュレーションロジックを「サーバーで描いて配信する版」と「ブラウザ単体で完結する版」の両方で持っているのは、デモ用途（配信サーバーなしでも動かせる）や、多人数が同時アクセスしたときの負荷分散の選択肢として理解できる。

:::message
📷 **画像プレースホルダー**：`/`（client.html・WebSocket版ビューア）と `/standalone.html`（ブラウザ単独版、`✓ DINOv2 Ready | ✓ SegHead Ready` のステータス表示が見えるもの）の2画面を並べたスクリーンショット。
:::

```js
// 旧 /client/ は /standalone.html へ301リダイレクトする後方互換対応
if(urlPath==='/client'||urlPath==='/client/'||urlPath==='/client/index.html'){
  res.writeHead(301,{'Location':'/standalone.html'}); res.end(); return;
}
```

URL体系を整理し直した際にも、古いURLを壊さず301リダイレクトで吸収している。小さなことだが、公開中のデモやブックマークを持つ利用者への配慮が随所に見える。

## 運用パラメータで負荷を調整する

サーバーは環境変数で挙動を細かく調整できる。

| 変数 | 既定 | 意味 |
|---|---|---|
| `WIDTH` / `HEIGHT` | 720 | 配信映像の解像度（メモリ/CPUに直結） |
| `FPS` | 30 | 配信フレームレート |
| `TICK` | 150 | シミュレーション1歩の間隔(ms) |
| `INFER_EVERY` | 10 | 何歩ごとに推論するか（上げると軽い） |
| `MAX_TRAIL` | 500 | 軌跡の最大点数 |
| `SEG_GATE` | 0 | 1でseg_headによる前進判定（既定はマップ配列） |

特に `INFER_EVERY` は象徴的なパラメータだ。DINOv2 + ポリシーの推論は決して軽くないため、**毎フレーム推論するのではなく、Nステップに1回だけ推論し、その間の行動はキャッシュされたactionを使い回す**という割り切りをしている（`prefetchAllActions` 関数、`server.js:406`）。

```js
async function prefetchAllActions(map, agents){
  if(stepCount % INFER_EVERY !== 0) return;
  for(const a of agents){
    actionCache[a.def.id] = await inferAction(map, a);
  }
}
```

「見る頻度」と「動く頻度」を意図的に分離することで、視覚認識のコストを抑えながら、体感上は滑らかに動いているように見せている。これも、理論上の理想（毎フレーム最新の観測で判断する）と、実際にCPU上で動かせる現実的な負荷のあいだの、明確なトレードオフの選択だ。

## この章までのまとめ

第3部を通して見てきたように、このリポジトリの実装は、第2部で描いた理想のアーキテクチャの全部を実装しているわけではない。VLMによる自然言語での状況説明もなければ、V-JEPA2による長期の物理予測もない。継続学習の仕組みもまだない。

その代わりに、**「DINOv2による視覚表現」「PPOによる軽量ポリシー学習」「ONNXによる学習/本番の橋渡し」「目標条件付けzによるLLM連携の足場」**という、比較的小さく閉じた4つの要素を、学習と本番で一貫性を保ちながら丁寧に作り込んでいる。理想のすべてを一度に実装しようとするのではなく、動くものを小さく確実に積み上げていく——この態度そのものが、次の第4部で扱うワールドモデルの理論を読むときにも、実装との距離感を測るための1つの参照点になるはずだ。
